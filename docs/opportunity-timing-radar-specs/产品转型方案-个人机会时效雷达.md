## 1. 最终定位

求职雷达不再定义为“岗位聚合”“岗位搜索”“岗位主动爬取工具”，也不把“官方链接”作为唯一卖点。

新的产品定位是：

> **当与你有关的求职机会发生时效变化时，职达告诉你：发生了什么、为什么与你有关、为什么现在值得行动。**

更口语化的产品承诺是：

> **不用每天刷招聘软件。职达每天替你看企业官网，只提醒真实、对口、仍有效、现在该处理的机会。**

产品的核心价值不是“有多少岗位”，而是把岗位库、官方源、保鲜、筛选、提醒和用户反馈组织成一个用户能处理的机会系统。

## 2. 为什么不是再做一个 BOSS / LinkedIn

### 2.1 市场已有能力

综合招聘平台已经覆盖了很多基础能力：

- BOSS 直聘所属的 Kanzhun 在官方投资者资料中长期强调其招聘平台连接求职者与企业用户，并依赖智能匹配、直聊和招聘服务变现；这类平台的优势是双边网络、雇主供给和即时沟通。
- LinkedIn 官方帮助文档已经支持职位推荐、每日/每周 job alerts、公司职位提醒，以及付费职位向合格候选人推送即时通知。

参考来源：

- [Kanzhun Annual Reports](https://ir.zhipin.com/financial-reports/annual-reports/)
- [Kanzhun Company Profile](https://ir.zhipin.com/about-us/company-profile/)
- [LinkedIn Job Alerts](https://www.linkedin.com/help/linkedin/answer/a511279)
- [LinkedIn Company Job Alerts](https://www.linkedin.com/help/linkedin/answer/a554166/create-job-alerts-for-specific-companies)
- [LinkedIn Job Recommendations](https://www.linkedin.com/help/linkedin/answer/a512279)
- [LinkedIn Instant Notifications for Paid Job Posts](https://www.linkedin.com/help/linkedin/answer/a414827/instant-notifications-for-paid-job-posts)

因此，求职雷达不能把以下能力当成核心差异化：

- 通用职位搜索；
- 推荐岗位；
- 收藏岗位；
- 订阅提醒；
- 公司关注；
- 简历匹配；
- 批量职位覆盖；
- 与 HR 沟通；
- 自动投递。

这些能力要么已经成熟，要么依赖强网络效应，要么会被通用 AI 工具商品化。

### 2.2 我们可切入的空位

现有平台的推荐和提醒大多服务于“平台内职位分发”，尤其会受到雇主发布、平台商业化和用户活跃机制影响。求职雷达的切入点不是替代它们，而是做一个更窄的候选人侧任务：

> **跨企业官方招聘源，持续监控与你有关的机会变化，并用低噪音方式告诉你今天该处理什么。**

这个任务的差异在于：

1. **官方源是证据层**  
   官方链接不是全部价值，但它能证明岗位来自企业官网或 ATS，而不是转载、中介、搬运或过期页面。

2. **新鲜度是信任层**  
   用户真正痛的是“岗位看起来还在，其实已经没了”。`last_seen_at`、source SLA、失活巡检和官方详情页质量门能构成“最近确认仍在招”的可信解释。

3. **机会变化是提醒层**  
   不提醒“所有岗位”，只提醒“与你有关且发生了值得行动的变化”：新出现、仍有效、截止临近、招聘动量、关闭/陈旧。

4. **个性化硬门是降噪层**  
   目标岗位、城市、阶段、学历、行业、排除词、关注公司共同决定是否进入机会队列。宁可少给，也不混入明显不相关岗位。

5. **行动闭环是长期资产**  
   用户保存、忽略、投递、反馈原因后，系统减少重复劳动，逐步形成个人机会模型。

## 3. 产品价值公式

新版求职雷达的价值公式：

```text
个人机会价值 =
  官方可信
  × 最近确认有效
  × 与用户目标匹配
  × 发生了值得行动的时效变化
  × 用户能一键处理并反馈
```

任一项缺失，产品价值都会下降：

- 只有官方可信：变成链接库；
- 只有最近确认有效：变成保鲜岗位库；
- 只有匹配：变成普通推荐；
- 只有变化：变成新闻流；
- 没有行动反馈：无法改善下一次机会队列。

所以产品核心不是“官方链接”，而是：

> **以官方源和保鲜机制为证据，把用户该处理的机会变化压缩成一个低噪音队列。**

## 4. 服务三类用户，但不做三个产品

用户已经明确希望同时服务三类人。正确方式不是做三套系统，而是一个机会引擎，三种模式。

### 4.1 冲刺找工作

用户状态：

- 未来 2–8 周积极投递；
- 每天愿意处理 5–20 个机会；
- 关注高匹配、仍有效、可立即投递；
- 对噪音容忍度低，但愿意看少量拓展机会。

核心体验：

- 每天打开“今日机会”；
- 看到新增、高匹配、截止临近；
- 处理动作是“值得投 / 不合适 / 已投递”；
- 强调“今天该处理什么”。

### 4.2 长期机会观察

用户状态：

- 当前不急着换工作；
- 有目标公司、方向或城市；
- 不想被大量职位打扰；
- 只关心异常好机会或关注公司明显变化。

核心体验：

- 低频摘要；
- 默认不展示普通相关岗位；
- 重点展示关注公司新开、仍在招、招聘动量变化；
- 强调“机会变化，不打扰”。

### 4.3 校招 / 实习窗口

用户状态：

- 对批次、截止、城市、岗位方向敏感；
- 企业开放和关闭窗口比单个岗位文案更重要；
- 需要知道“哪些新批次刚开”“哪些快截止”。

核心体验：

- 阶段硬门优先；
- 截止提醒优先级高；
- 公司批量开放作为机会信号；
- 强调“窗口期不要错过”。

### 4.4 共用能力

三类用户共用：

- 官方岗位库；
- 保鲜机制；
- 个性化画像；
- 五类时效信号；
- 机会队列；
- 用户动作与反馈；
- 管理员覆盖与数据质量后台。

区别只体现在：

- 召回阈值；
- 排序权重；
- 提醒频率；
- UI 文案；
- 空状态；
- 分区优先级。

## 5. V1 对用户承诺的五类信号

V1 只承诺能用现有数据或低成本补充数据可靠支撑的信号。

| 信号 | 用户问题 | 可用证据 | V1 是否展示 |
|---|---|---|---|
| 新机会 | 有什么刚出现、适合我的岗位？ | `first_seen_at` + 匹配硬门 + score | 是 |
| 仍在招 | 这个岗位现在还有效吗？ | `last_seen_at` + source `crawl_method` SLA | 是 |
| 截止临近 | 哪些机会快来不及了？ | `deadline` 可解析日期 | 是 |
| 招聘动量 | 哪些关注公司/方向最近明显在招？ | 近 14/30 天 `first_seen_at` 聚合 | 是，带样本门槛 |
| 关闭/陈旧 | 我关注的机会是不是已经不值得投？ | `status` + freshness stale | 是，主要用于已保存/关注机会 |

V1 不展示、也不作为卖点：

| 信号 | 原因 |
|---|---|
| 岗位内容变更 | 发生频率低，噪声高，用户行动价值弱；先只做后台埋点 |
| 重新开放 | 当前 `expired` sticky 语义会影响可靠识别，需要事件历史后再做 |
| 全市场最早发现 | 无法证明，也不需要作为价值承诺 |
| BOSS/LinkedIn 没有的岗位 | 未系统性抓取竞品平台，不能承诺 |
| 公司长期大小年 | 需要一年以上稳定历史，当前只能做短期招聘动量 |

## 6. 岗位内容变更的处理决策

岗位内容变更不进入 V1 用户价值。

原因：

1. **低频**：多数企业 JD 发布后不会频繁改动，真正常见变化是新发、下架、截止和仍在招。
2. **高噪声**：页面脚本、推荐位、页脚、格式、时间戳、ATS 模板变化都可能导致 hash 变化。
3. **低行动价值**：用户通常不关心 JD 文案改了几句话，除非薪资、地点、要求发生结构化变化；当前数据不具备稳定字段级 diff。
4. **实现成本不低**：需要历史快照、字段规范化、噪声过滤、变更归因和误报控制。

正确处理：

- 保留 `content_hash`；
- 在事件台账阶段记录 `JOB_CONTENT_HASH_CHANGED_INTERNAL`；
- 不对用户展示“岗位内容变更”；
- 不在报名帖、Landing、核心文案里宣传此能力；
- 等未来能稳定提取薪资、地点、学历、阶段、技能要求等结构化字段后，再考虑“重要字段变更”。

## 7. 主动爬岗位功能的处理决策

普通用户主动爬岗位应降级，不作为主入口。

保留原因：

- 后台保鲜需要；
- 管理员扩源需要；
- 用户关注但未覆盖公司时，需要形成接入请求；
- 数据异常诊断需要；
- 已经完成的 crawler 基础设施不应被破坏性删除。

降级方式：

- `/jobs` 不再把“刷新对口公司 / 发掘新公司”作为普通用户主动作；
- 用户添加关注公司后，只看到“已覆盖 / 排队接入 / 研究中 / 暂不支持”；
- 用户不等待爬虫，不看 workflow、source、crawl_run、jobs_created；
- 管理员后台处理覆盖请求；
- feature flag 保留手动刷新入口，默认关闭。

原则：

> 爬虫是基础设施，不是产品体验。

## 8. 技术可行性判断

### 8.1 当前已足够支撑的能力

现有 `jobs` 表和 crawler 语义已经支持：

- `first_seen_at`：识别新发现机会；
- `last_seen_at`：识别最近确认仍在招；
- `status`：识别 active / removed / expired / error；
- `content_hash`：为未来内容变化埋点；
- `deadline`：在可解析时支撑截止提醒；
- `source_id`：关联 source crawl method、enabled 状态、公司；
- `summary`：支撑 JD 有效性门槛；
- `company/title/location/job_type/education/experience`：支撑匹配和分区；
- `job_actions`：支撑保存、忽略、投递、浏览；
- `events`：支撑轻量产品埋点；
- 独立 jobs PostgreSQL + Supabase 用户库：已形成清晰读写边界。

因此，第一版“个人机会时效雷达”不是从 0 开始。它是把已有供给、保鲜和匹配能力重新组织成用户可感知的机会变化。

### 8.2 当前不足，需要补齐的能力

为了让长期产品叙事更稳，需要补齐：

1. **用户模式**  
   `sprint / watch / campus` 三种模式，影响阈值、分区和提醒频率。

2. **显式信号类型**  
   不只返回 jobs，而是返回 `signal_type`：新机会、仍在招、截止临近、招聘动量、关闭/陈旧。

3. **事件台账**  
   `jobs` 当前是 current state 表，不是历史事件表。长期要有 append-only `job_events`，记录首次发现、确认仍在招、关闭、内容 hash 变化等事实。

4. **用户投递/消费台账**  
   要知道某个机会变化是否已经给某个用户展示、是否处理、是否再次提醒，避免重复打扰。

5. **截止日期解析**  
   当前 `deadline` 是文本，只有能解析成日期时才可触发 `DEADLINE_SOON`。

6. **招聘动量阈值**  
   必须有样本门槛，避免用 1–2 条岗位制造趋势。

### 8.3 不应承诺的能力

当前不能承诺：

- 全网覆盖；
- 竞品没有；
- 最早发现；
- 自动投递；
- 内容字段精确变更；
- 重新开放可靠识别；
- 长周期招聘趋势；
- Offer 或面试结果提升；
- 岗位 100% 有效。

产品对外表述必须坚持：

> **官方源、最近确认、低噪音、与你有关、现在值得处理。**

不要写成：

> “全网最快、所有岗位、永不过期、自动帮你找工作。”

## 9. 产品结构

新版产品由四层组成：

```text
官方岗位供给层
  crawler / source / jobs / enrich / liveness

机会事实层
  first_seen / last_seen / status / deadline / source SLA / company momentum

个人筛选层
  mode / target role / city / company / stage / education / industry / exclusions

行动交付层
  today feed / digest / save / ignore / applied / feedback / admin metrics
```

用户只应看到第四层，以及必要的事实解释。前三层只在管理员和技术文档中出现。

## 11. 北极星指标

新版北极星指标：

> **每周每活跃用户处理的有效机会数。**

“有效机会”必须同时满足：

- 来自官方源；
- `status='active'`；
- 最近确认仍在招或处于明确可解释的 aging；
- 通过用户画像硬门；
- 具有至少一个时效信号；
- 被用户处理。

“处理”包括：

- 标记值得投；
- 打开官方链接；
- 标记已投递；
- 忽略并选择原因；
- 关闭已失效提醒。

辅助指标：

- Feed 打开率；
- Feed 中机会处理率；
- 忽略原因分布；
- 官方链接打开率；
- 已投递转化率；
- `ignored` 后同类岗位减少情况；
- 关注公司覆盖率；
- 机会 freshness 分布；
- 召回 cap 触发率；
- email digest 退订率。

不作为核心指标：

- 总岗位数；
- crawler 次数；
- jobs_created；
- source 数；
- 搜索次数；
- 单次页面浏览量。

## 12. 创业验证重点

接下来不是继续堆供给，而是验证用户是否愿意因为“机会时效雷达”持续回来。

需要验证的核心问题：

1. 用户是否觉得“今天机会”比自己刷平台更省时间？
2. 用户是否相信“最近确认仍在招”的解释？
3. 用户是否愿意认真填写目标和模式？
4. 用户是否愿意对不合适岗位给原因？
5. 长期观察用户是否接受低频提醒？
6. 校招/实习用户是否认为截止和批次提醒有价值？
7. 官方链接是否提升投递信心？
8. 当列表很少时，用户是否接受“宁缺毋滥”？

如果以上不能成立，继续扩 source 也无法解决产品价值问题。

## 13. 开发投入顺序

不按“90 天路线图”写，而按依赖顺序写。

### 13.1 第一优先级：兑现核心体验

- 用户选择模式；
- `Today` 变成唯一主产品；
- 五类信号进入 feed；
- 卡片说明“为什么现在”；
- 普通用户不再主动爬；
- 动作反馈闭环；
- 真实指标埋点。

### 13.2 第二优先级：让系统可长期运行

- 事件台账；
- 用户机会投递/展示去重；
- 招聘动量稳定阈值；
- 管理员处理关注公司；
- source freshness 监控；
- digest 发送和退订。

### 13.3 第三优先级：增强差异化

- 更细的公司招聘节奏；
- 学校/专业/城市校招窗口；
- 重要字段变更；
- 求职目标组合；
- 高校/社群/职业咨询侧协作。

## 14. 最终产品判断

这个项目仍然有创业价值，但前提是不要把它做成“另一个岗位库”。

更准确的判断是：

- 岗位搜索是入口，不是主产品；
- 官方链接是证据，不是唯一优势；
- 爬虫是基础设施，不是用户价值；
- AI 匹配是辅助，不是壁垒；
- 机会时效和低噪音行动队列才是主叙事。

新版方向成立的关键是：

> **用户每次打开职达，都能少想一点、少筛一点、少踩一个失效岗位，并更清楚今天该投什么。**
