# 05 验收规格 v3

> 日期：2026-06-24
> 地基：`产品方向v2-岗位保鲜雷达.md`（v3）+ `01`~`04` 四份 spec
> 取代：`2026-06-24-opportunity-timing-radar-acceptance-spec.md`（Codex 三模式版）。
> 性质：实现 agent 与验收 agent 共用。**只认真实行为，不认表面 UI；不用 mock 冒充成功。**

---

## 0. 验收原则

- API 返回真实字段；数据库读回真实写入；
- today 每张卡至少一个 signal + 官方链接 + 最近核验时间或"待确认"；
- **不吹绝对"没有死岗"**，对外用"刚核验、极少踩空"；
- 不展示岗位内容变更/重新开放/全网最快/BOSS 没有/自动投递；
- 不把 crawler 运行次数/source 数当用户价值；
- **没有 sprint/watch/campus 三模式**——身份来自 `experience_stage`，强度是 active/passive 自调。

## 1. 文案一致性

```bash
rg -n "没有死岗|零死岗|全网最快|全网最早|BOSS.*没有|自动投递|岗位内容变更|JD.*变更|重新开放|刚发布|全网首发" app components lib
```

允许：docs 中的"不展示/不承诺"说明、测试中的禁止性校验、内部事件常量。
不允许：Landing/today 卡片/报名帖出现上述绝对或夸大表述；`STILL_OPEN` 写成"刚发布"；仅有 `first_seen_at`（无官方 posted_at）时出现"刚发布/我们新发现"。

允许的核验表述：`最近确认仍在招`、`刚核验`、`待确认`、`我们新发现的官网机会`（仅当官方 posted_at 可用）。

## 2. 数据模型验收

### 2.1 强度（不是模式）

```sql
select column_name from information_schema.columns
where table_name='user_preferences'
  and column_name in ('radar_intensity','radar_intensity_source','radar_intensity_updated_at');
```
必须返回三行。约束：`radar_intensity ∈ (active,passive)` 默认 active；`radar_intensity_source ∈ (default,user,auto)`。

**反向校验**：库中**不应存在** `user_preferences.radar_mode` 列（确认没走回三模式老路）。

### 2.2 身份沿用既有

`candidate_profiles.experience_stage` 存在（迁移 006），值域 `"" / 实习 / 校招 / 社招`。

### 2.3 jobs-db：确认下架时间 + 事件表

```sql
-- 在香港 jobs 库执行
select column_name from information_schema.columns where table_name='jobs' and column_name='confirmed_closed_at';  -- 必有
select to_regclass('public.job_events');  -- 非空
```
`job_events.event_type` 只允许：`FIRST_SEEN / OFFICIAL_POSTED / CONFIRMED_OPEN / CLOSED / REAPPEARED`。
若该阶段未做事件表，验收报告必须写明"事件台账未执行；V1 用户侧不依赖；'我们新发现/动量' 不展示"。

## 3. API 验收

### 3.1 `/api/preferences`
- GET 返回 `radar_intensity` + `radar_intensity_source`；未登录 401；不泄漏他人数据；不返回 source 技术细节。
- PUT：带 `user_id` → 400 `validation_failed`；`radar_intensity='passive'` 写入后 `radar_intensity_source='user'`；daily_limit 按强度 clamp；保存 target_companies 同步 `company_watch_requests`。

### 3.2 `/api/opportunities`
返回含 `intensity`、`stage`、`counts.by_signal`、动态 `sections`。每个 opportunity 含 `job/score/tier/reasons/signals/freshness/lastCheckedAt/officialPostedAt/deadlineAt/intensity`，且 `signals.length ≥ 1`。
禁止：mock opportunity；feed 失败仍 `ok:true`；无画像返回随机岗；`enrich_checked_at` 缺失却显示"最近确认仍在招"。

### 3.3 `/api/radar/open`
更新 `user_radar_state.last_opened_at`；写 `radar_feed_opened`；不接受 user_id；幂等；不创建 job action。

### 3.4 `/api/job-actions/[jobId]`
ignored 无 reason_code → 400；有 reason_code → 成功且读回 `reason_code` + `job_snapshot` 非空；saved/applied 清空 reason；同岗主动作唯一；job 不存在 → 404 `job_not_found`；客户端传 `job_snapshot`/`user_id` → 400。

### 3.5 错误契约
反例对照（节选）：

| 请求 | 预期 |
|---|---|
| 未登录 `GET /api/opportunities` | 401 `unauthorized` |
| `PUT /api/preferences` 带 `user_id` | 400 `validation_failed` |
| `PUT /api/preferences` `radar_intensity='x'` | 400 `validation_failed` |
| jobs store 抛错后 `GET /api/opportunities` | 503 `feed_unavailable`，**无 mock** |
| `POST /api/job-actions/:id` ignored 无 reason | 400 `validation_failed` |

错误响应不得含堆栈/service role 原文/邮箱/简历/完整 JD/reason_text/带 token URL。

## 4. 强度（自调）验收

1. **手动优先**：用户 PUT `radar_intensity='passive'` → feed 的 `intensity='passive'`、daily_limit 收窄、不推拓展区。
2. **行为自调**：构造一个 30 天前手动设、近 14 天无动作、长期未打开的用户 → `resolveIntensity` 返回 `passive`（`source='auto'`）；构造近 3 天打开 + 有动作 → `active`。
3. **强度不卡门**：强度无论 active/passive，都不改变 `profile_ready`。
4. **关键提醒不被压**：passive 用户，其 saved 岗快截止/关闭仍出现在关键提醒区（不被强度裁掉）。

---

## 5. 分层核验 SLA 验收（对接 01 spec）

1. `enrich_checked_at` 为 26h 前的 active 岗 → 不进 today 主清单；同岗在 `/api/jobs/search` 出现但标"待确认"。
2. `enrich_checked_at IS NULL` 的岗 → `meetsVerifyTier(...,"today").ok=false`，不写"仍在招"。
3. **点击有效率可算**：触发若干 `opportunity_official_opened` + `job_liveness_at_click`，能从 `events` 聚合出有效率，并在 admin/health 显示（目标 ≥99%，按 adapter 拆分）。
4. **点击不卡**：点击官网，可探源死岗弹提示但仍可打开、活岗/未知秒开（无 5–8s 阻塞）。
5. **SPA 新岗优先**：近 48h 新增的 SPA 岗在 `audit_dead_links --prioritize-new` 中排队头（SQL 读回证明）。

## 6. 信号（标签）验收

1. **STILL_OPEN**：active + `enrich_checked_at ≤24h` + 匹配 → signals 含 STILL_OPEN，卡片显示"最近确认仍在招"；`enrich_checked_at` 缺失则不触发。
2. **DEADLINE_SOON**：deadline `2026-07-01` + now `2026-06-28` + 社招 → 触发；`长期有效` → 不触发；校招 14 天窗内触发、15 天不触发。
3. **CLOSED_OR_STALE**：saved 岗变 expired → 进 saved/状态区"可能已关闭"，不进主清单；普通未交互 stale 岗不进 today。
4. **NEWLY_DISCOVERED**：仅 `posted_at`（官方）可用且在窗口时触发；只有 `first_seen_at` → **不触发**、不出现"刚发布"。
5. **COMPANY_MOMENTUM**：不出现在 C 端"猛招"文案；新接源/重灌不算动量（防假动量守则，见 02 spec §4.3）。
6. **content_hash 变化**：不产生任何用户侧 signal；today 不展示"内容变更"。

## 7. 假成功 / Mock 反例验收

### 7.1 无画像不返回岗位
新建无 `user_preferences`/`candidate_profiles` 用户 → `GET /api/opportunities`：`profile_ready=false`、`sections=[]` 或全空、`counts.total=0`、**不返回任何岗位**。失败判定：返回固定示例/最新 active/mock 公司名。

### 7.2 jobs store 故障不假成功
mock `recallOpportunityCandidates` 抛错 → HTTP 503、`ok:false`、`error.code='feed_unavailable'`、不写任何台账。失败判定：200 空 feed 但 `ok:true` / 返回 hardcoded opportunity。

### 7.3 官方链接真实性
对每个返回的 opportunity：`job.id` 能在权威库读回；`jd_url` 非空；主清单里 `status='active'`（关闭岗只在状态区）；`summary` trim ≥60（状态区除外）；**不得出现 `https://example.com/job` 等测试常量**。

### 7.4 前端假成功回滚
让 `POST /api/job-actions/:id` 返回 500 → 前端展示失败 + 本地回滚 + 不写 `job_actions` + 不写成功事件。失败判定：按钮显示成功/卡片消失/刷新后不一致。

## 8. UI 验收

- **Landing**：表达 官网 / 刚核验仍在招 / 适合你 / 少踩死岗 / 少筛少烦；**不**表达 主动爬虫 / 全网最全 / 内容变更 / 自动投递 / BOSS 没有 / 没有死岗(绝对)。
- **Today**：登录后默认页；显示身份 + 强度；signal counts；每卡有 signal label + 官方链接 + 最近核验时间或"待确认"；有 saved/ignored/applied；ignored 选原因；**不**展示 score 数字 / crawler / source / workflow；无画像不展示岗。
- **Preferences**：身份 + 强度开关（active/passive）可设并 API 读回；显示关注公司覆盖状态；普通用户看不到"启动爬虫"。
- **Jobs**：保留搜索/筛选/保存；普通用户默认不见主动 refresh/discovery（admin/flag 可留）；不污染 today 主叙事。

## 11. 最终判定

**可通过**：
1. 无三模式；身份来自 experience_stage、强度 active/passive 可手动 + 自调且不卡门、不压关键提醒；
2. today 是机会清单不是岗位列表，每卡有 signal + 官方链接 + 最近核验时间/待确认；
3. 分层核验 SLA 生效；`enrich_checked_at` 缺失/超时不冒充"仍在招"；
4. 点击有效率可度量并展示；点击不被卡 5–8s；
5. 信号当标签：STILL_OPEN 主力、DEADLINE/CLOSED 可构造、NEWLY_DISCOVERED 仅官方 posted_at 时、动量不上 C 端；
6. ignored 收 reason_code；无画像不展示岗；feed 失败 503 不假成功；
7. 普通用户主动爬取入口降级；
8. 三不变量（expired sticky / preserve-if-empty / enrich_checked_at 不被覆盖）未破；
9. 测试命令通过或失败原因明确且非本改动引入。

**不可通过**：
- 出现 sprint/watch/campus 模式或 `radar_mode` 列；
- 对外出现"没有死岗(绝对)/全网最快/BOSS 没有/刚发布(仅 first_seen)"；
- mock feed 冒充真实；无画像仍展示岗；today 卡无 signal；
- 把核验做成阻塞点击的长等待；
- ignored 无 reason 仍成功；action 写入绕过当前用户；feed 失败仍 `ok:true`；
- 破坏任一回归不变量。
