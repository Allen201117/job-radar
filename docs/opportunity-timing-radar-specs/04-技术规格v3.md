## 1. 数据模型变更

### 1.1 Supabase 迁移 `164_radar_intensity.sql`

```sql
-- 用户强度（手动初值 + 手动覆盖标记）
alter table user_preferences
  add column if not exists radar_intensity text not null default 'active'
    check (radar_intensity in ('active','passive'));
alter table user_preferences
  add column if not exists radar_intensity_source text not null default 'default'
    check (radar_intensity_source in ('default','user','auto'));
alter table user_preferences
  add column if not exists radar_intensity_updated_at timestamptz;
```

要求：幂等；不接受客户端 `user_id`；只写当前用户；RLS 不放宽（沿用 user_preferences 现有策略）。

`radar_intensity_source` 含义：`default`=系统默认 / `user`=用户手动设 / `auto`=行为自调写入。优先级：近期 `user` > `auto` > `default`（见 §3）。

> 行为自调所需的"最近打开/动作"信号，复用 `user_radar_state`（迁移 161：`last_opened_at` 等）与 `job_actions`（迁移 162），**不新增大表**。若需缓存自调结果，写回 `user_preferences.radar_intensity`（`radar_intensity_source='auto'`）。

### 1.2 jobs-db 变更（香港 PG，走 `jobs-db/schema.sql`）

1. 厘清 `posted_at` 语义 = **官方发布时间**（来源：结构化数据/接口/adapter，拿不到为 NULL），见 02 spec §4。
2. 新增列：
   ```sql
   alter table jobs add column if not exists confirmed_closed_at timestamptz;  -- 判死时刻
   ```
   写入：`enrich_backlog.py`/`audit_dead_links.py`/`liveness-check` 把岗置 `expired` 时一并写 `confirmed_closed_at=now()`（best-effort）。
3. 新增 `job_events`（append-only 里程碑表，DDL 与 `event_key` 规则见 02 spec §5.1）。**不做 RLS**（在 jobs-db，不直供客户端读）。

### 1.3 明确不新增

`radar_mode` 列；`user_opportunity_deliveries` 大表（V1 用 `user_radar_state` + 7 天去重逻辑即可，delivery ledger 留到时效信号成熟后再评估）；向量库 / Redis / 队列 / per-user feed cache 大表 / JD 全量历史快照。

## 2. 类型（扩展 `lib/opportunities/types.ts`）

现有：`ExperienceStage`、`RadarProfile`、`FreshnessState`、`MatchFacts`、`EligibilityResult`、`OpportunityTier`、`Opportunity`、`OpportunityFeed`、`FeedSections`。**保留，扩展如下**：

```ts
export type RadarIntensity = "active" | "passive";

export type VerifyTier = "today" | "search" | "admin";

// 信号当“标签”，不是入选唯一理由
export type OpportunitySignalType =
  | "STILL_OPEN"          // 最近确认仍在招（护城河外显，主力）
  | "DEADLINE_SOON"       // 快截止
  | "CLOSED_OR_STALE"     // 收藏/关注对象关闭或陈旧（关键提醒）
  | "CAMPUS_WINDOW"       // 校招通道开放（校招季强化）
  | "NEWLY_DISCOVERED"    // 我们新发现（需官方 posted_at；当前 first_seen 污染下不可用）
  | "COMPANY_MOMENTUM";   // 公司动量（B 端/admin，不上 C 端“猛招”）

export interface OpportunitySignal {
  type: OpportunitySignalType;
  label: string;          // 展示文案（受 03 spec 文案约束）
  priority: number;
  isCritical: boolean;    // true=关键提醒（不受强度压制）
  evidence: {
    lastCheckedAt?: string | null;   // = jobs.enrich_checked_at
    officialPostedAt?: string | null;// = jobs.posted_at（官方发布；可空）
    deadlineAt?: string | null;
    status?: string | null;
    freshness?: FreshnessState;
  };
}
```

`Opportunity` 增 `signals: OpportunitySignal[]`（≥1）、`intensity: RadarIntensity`、`lastCheckedAt`、`officialPostedAt`、`deadlineAt`。`OpportunityFeed` 增 `intensity`、`stage`、`counts.by_signal`、分区按身份×强度×已触发信号动态生成。

## 3. 强度推导（新增 `lib/opportunities/intensity.ts`）

```ts
export function resolveIntensity(input: {
  manual: RadarIntensity | null;          // user_preferences.radar_intensity（source='user'）
  manualAgeDays: number | null;           // 手动设置距今天数
  lastOpenedAt: string | null;            // user_radar_state
  recentActionCount14d: number;           // job_actions 近14天动作数
  hasTargetCompanies: boolean;
  now: Date;
}): { intensity: RadarIntensity; source: "user" | "auto" | "default" };
```

规则（纯函数，可测）：
1. 用户近期（如 ≤30 天）手动设过 → 用手动值（`source='user'`）；
2. 否则行为自调（`source='auto'`）：近 14 天有动作或近 3 天打开过 → `active`；长期（如 >14 天）未打开 → `passive`；
3. 新用户无信号 → 默认 `active`（蜜月期，`source='default'`）。

强度只影响 `daily_limit`、分区取舍、推送频率、入选门槛（见 §6），**不影响 `profile_ready`、不裁剪关键提醒**。

## 4. Profile readiness（⚠️ 必改：城市改为非硬门）

**与现有代码不同，是必改项**：现有 `profileReadiness()`（`lib/opportunities/profile.ts`）是 `ready = hasContent && hasLocation`——**城市永远是硬门**。v3 改为：

- `ready = hasContent`（targetRoles/keywords/companies 任一即可）；
- **城市不再是 readiness 硬门**：缺城市照样 ready，只是不按城市过滤、卡片标"城市未限定"（服务"只想盯几家公司、没指定城市"的观望用户）；
- **身份、强度都不参与 readiness**（身份只在有岗位时做资格过滤；强度只调推荐量）。
- 阶段资格门用 `eligibility.ts` 的 `stageState`（严格同阶段，矩阵见 03 spec §2.1），与 readiness 解耦。

`profileReadiness()` 返回 `{ ready, missingContent, missingLocation }` 保留；`ready` 去掉 `&& hasLocation`，`missingLocation` 仅供"建议补城市"软提示、不阻断。

`dailyLimit` 按强度 clamp（active 偏多 / passive 偏少，沿用 5–30 范围）。

## 5. 新鲜度与分层核验（对接 01 spec）

- 保留 `lib/opportunities/freshness.ts` 的 `freshnessState(lastSeenAt, crawlMethod, now)`（按 crawl_method SLA：http 18/36h、playwright 36/72h、manual 72/144h + `last_seen_at` → verified/aging/stale/unknown，源侧抓取新鲜度）。
- **新增** `meetsVerifyTier(job, tier, now)`（签名见 01 spec §2.2），叠加"逐岗核验新鲜度"（按 `enrich_checked_at` 年龄）：
  - `today` tier：`enrich_checked_at` ≤24h 且非 NULL；
  - `search` tier：≤72h，否则展示但标"待确认"；
  - `admin` tier：无时限，禁写"仍在招/刚确认"。
- `buildOpportunityFeed()` 硬门阶段对 today 主清单要求 `meetsVerifyTier(job,"today").ok`。

## 6. 信号派生（降级为标签，`lib/opportunities/signals.ts`）

`deriveOpportunitySignals(job, facts, profile, parsedDeadline, now)` 返回 `OpportunitySignal[]`，优先级顺序：

1. **CLOSED_OR_STALE**（`isCritical=true`）：仅对 saved/viewed/关注公司/曾提醒对象；status∈expired/removed/error 或 freshness stale。
2. **DEADLINE_SOON**：`parseDeadline` 出明确日期且在窗口（社招/实习 7 天、校招 14 天）；active；freshness verified/aging。校招身份下置 `isCritical=true`。
3. **STILL_OPEN**：active + `meetsVerifyTier(today)` + 非关闭。**主力标签**。
4. **CAMPUS_WINDOW**：关注公司近期新增校招/实习岗（带样本门槛）；校招季强化，先简单。
5. **NEWLY_DISCOVERED**：**仅当 `job.posted_at`（官方发布）非空且在窗口内**；只有 `first_seen_at` 时**不产生**（当前 6/15 重建污染，见 02 spec §4.2 解禁条件）。文案禁"刚发布/全网首发"。
6. **COMPANY_MOMENTUM**：公司级，受 02 spec §4.3 防假动量守则约束；**不映射 C 端"猛招"**，先供 admin/高校看板。

约束：
- `enrich_checked_at` 缺失 → 不产生 STILL_OPEN/NEWLY_DISCOVERED；
- `content_hash` 变化**不产生任何用户侧 signal**（内部事件 §1.2/02 spec §5）；
- 每个进 today 的 opportunity `signals.length ≥ 1`；signal 不足不补无关岗。

### 6.1 Deadline 解析（`lib/opportunities/deadline.ts`）

```ts
export function parseDeadline(raw: string|null, now: Date): { date: string; confidence: "high"|"medium" } | null;
```
解析：`YYYY-MM-DD`、`YYYY/MM/DD`、`YYYY.MM.DD`、`YYYY年M月D日`、`M月D日`（按 now 推年，已过 30 天则取下一年，medium）。不解析：`长期有效`/`招满即止`/`尽快`/无法判断的多日期。

## 7. 分区（`lib/opportunities/grouping.ts`，按身份×强度×已触发信号）

**不用固定三模式布局**。`groupOpportunities()` 输入 `(opportunities, stage, intensity)`，输出动态有序 sections：

- **关键提醒区**永远置顶（含 CLOSED_OR_STALE / DEADLINE_SOON(校招) / CAMPUS_WINDOW）；
- **主清单**：STILL_OPEN 的对口机会；
- **拓展看看**：仅 `intensity='active'`；
- **动量**：仅数据可信且非 C 端"猛招"表述；
- **等待再次确认**：aging 等。

截断：总量 ≤ `daily_limit`（关键提醒不计入裁剪、不被截）；一个 job 只出现一次（按 signal 优先级归一）；saved/applied/ignored 不进主清单。

## 8. API 规格

### 8.0 通用契约（沿用 Codex 版，保留）

成功 `{ "ok": true, ... }`；失败 `{ "ok": false, "error": { "code", "message", "fields"? } }`。`code` snake_case；不返回堆栈/service role 原文/邮箱/简历/完整 JD/reason_text/带 token URL。

状态码矩阵：

| HTTP | code | 场景 |
|---:|---|---|
| 400 | `invalid_json` / `validation_failed` | body 非 JSON / 字段非法、含 `user_id`、数组超限 |
| 401 | `unauthorized` | 未登录 |
| 403 | `forbidden` | 越权 |
| 404 | `job_not_found` | jobId 在权威库不存在 |
| 409 | `conflict` | 主动作 RPC 冲突 |
| 422 | `profile_not_ready` | 需完整画像的写入动作但身份字段不足 |
| 503 | `feed_unavailable` / `preferences_unavailable` / `radar_state_unavailable` / `job_action_unavailable` | 后端读写失败，**不返回 mock** |

鉴权：用户 API 走 `requireUser()`，admin 走 `requireAdmin()`；请求体含 `user_id` → 400；一律用服务端当前用户 id。

### 8.1 `/api/preferences`（已存在，扩展）

- GET 返回新增 `radar_intensity`、`radar_intensity_source`；其余沿用（target_*, daily_limit, profile_ready, coverage, notification_settings）。
- PUT 接受 `radar_intensity`（写 `radar_intensity_source='user'` + `radar_intensity_updated_at=now()`）；其余校验沿用（数组 trim/去空/去重、≤30 项、单项 ≤80 字、daily_limit 按强度 clamp、不接受 user_id、保存 target_companies 同步 `company_watch_requests`）。

### 8.2 `/api/opportunities`（已存在，扩展）

- 返回 `intensity`、`stage`、`counts.by_signal`、动态 `sections`，每个 opportunity `signals.length≥1` 且含 `lastCheckedAt`/`officialPostedAt`/`deadlineAt`。
- GET 不更新 `last_opened_at`；profile 不 ready 返回 200 + 空 sections（`profile_ready=false`，不是错误）；
- 失败返回 503 `feed_unavailable`，**不返回 mock、不兜底最新岗**；
- 每个 opportunity 的 `job.id` 必须能从权威 jobs 库读回。

### 8.3 `/api/radar/open`（已存在）

更新 `user_radar_state.last_opened_at`、写 `radar_feed_opened` event；不接受 user_id；不创建 job action；幂等。

### 8.4 `/api/job-actions/[jobId]`（已存在，迁移 162 + RPC）

主动作互斥；ignored 必须 `reason_code`（白名单，见 03 spec §6）；`reason_text` ≤200；服务端用 `jobsByIds` 校验存在并生成 snapshot（不接受客户端 snapshot）；saved/applied 清空 reason；失败前端回滚不乐观永久改变。

### 8.5 点击/核验相关（新增，见 01 spec）

- `/api/jobs/liveness-check`（已存在，**批量展示核验**）：扩展覆盖 SPA 源"轻量信号"探活（不在请求里起无头浏览器）；返回该批 dead id 供看板隐藏。
- **`POST /api/jobs/[jobId]/liveness`（新增，单岗点击核验，契约见 01 spec §4.4）**：返回 `{ ok, result: alive|dead|unknown }`，封顶 2.5s，探不动返回 unknown；判死写 `expired`+`confirmed_closed_at`、判活 touch；顺带打 `job_liveness_at_click`。**与批量端点区别：单岗、返回三态、供点击用。**
- 点击官网链接：前端走点击前临门核验（封顶 2.5s、**判死只提示、默认放行**），并打 `opportunity_official_opened` + `job_liveness_at_click`（见 01 spec §5）。

### 11.1 job_events 分期与"可通过"口径（消除与 05 spec 的冲突）

- `job_events`（+ 结构化数据取官方 `posted_at`）是 **Phase 3、可延期**；Phase 1+2 不依赖它，可独立验收通过。
- **延期时**：`NEWLY_DISCOVERED` / `COMPANY_MOMENTUM` 两个信号**不得上线**（缺真实时间线会瞎报）；验收报告写明"事件台账未执行，对应信号未上"。
- **若要上线 `NEWLY_DISCOVERED` 或 `COMPANY_MOMENTUM`**：`job_events` + 官方 `posted_at` 解析 + 防假动量守则（02 spec §4.3）**必须同时具备**，缺则该信号验收不可通过。
