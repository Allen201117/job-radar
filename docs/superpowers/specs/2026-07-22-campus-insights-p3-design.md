## 0.5 决策定稿（2026-07-22，A0 live 摸底后 → 已 brainstorm 拍板，本段优先于下文草案表述）

**接手 session 已完成 §9 的 A0 前置摸底（直连生产香港库 + httpx 探官方页），并据真数据与创始人拍板。以下为定稿，与下文草案冲突时以本段为准。**

### A0 实测结论（生产库真数据，非猜测）

快路①（自有岗位派生）现实能力，比草案设想骨感：
- ✅ **每公司校招总 HC 体量**可靠：活跃校招岗（非实习）12616 个 / 474 家公司。
- ❌ **提前批 vs 正式批 HC 硬差异做不到**：岗位标题/`job_type` 里几乎没有批次词（全库 `title ~ '正式批'` 命中 **0**、提前批 43、补录 24）。批次是**招聘周期层**概念（公司整体宣布「8 月开提前批」），**不落在单个岗位上** → 无法从自有岗位按批次拆 HC。
- ❌ **科技大厂今年精确截止日期覆盖 = 0**：字节/腾讯/美团/阿里/小米/网易/京东/百度/快手 9 家核心大厂，**可用 deadline 全为 0**（字节 66 个有 deadline 的全是「长期有效」占位）。目标用户最想投的公司，自有库给不出精确日期。
- ⚠️ **传统制造/央企 deadline 反而 100% 且真实**：郑州宇通 `2026-08-01`（345/345）、中国一汽、中广核、国机、五矿、TCL、长城、伊利…——**但这些不是目标用户（科技/新经济/消费）关心的公司**。
- ⚠️ **deadline 是 text 列且混大量垃圾占位**：`长期有效`/`3000-01-01`/`2070-11-29`/`2042-01-31` 等，用前必须清洗成「近未来真实日期」。整体覆盖仅 19.3%。
- ❌ **first_seen_at 不可用于算批次开放时间**：被 2026-06-15 库重建污染（几乎全部 first_seen_min 落在 06-15/16/17），非真实首现。

快路②（官方校招页 httpx 直取）现实能力，混合：
- ✅ **部分大厂官方页 SSR 有数据可抽**：`jobs.bytedance.com/campus` 返回 819KB HTML，含「投递时间/网申」+ 日期（2026-03/04/05）→ 可 httpx 直取 + LLM 抽。
- ❌ **部分大厂官方页是 SPA 空壳**：`join.qq.com/campus.html`(1.7KB)、`talent.baidu.com/.../campus.html`(2.8KB) → httpx 拿不到内容，需逐家啃 JSON 接口。

**核心张力**：目标用户最想要的「字节今年几号截止」→ 自有库 = 0，只能靠快路② 啃官方页；而官方页一半是 SSR 能抽、一半是 SPA 空壳。**草案把快路① 当「最快最该先做、能扛精确日期 + HC 差异大部分」对目标用户不成立** → 主次重排。

### 拍板决策（创始人，2026-07-22）

1. **精确日期主力 = 快路② 通用官方页抽取器 + 诚实留白**（不逐家 bespoke SPA）：写一个通用「抓官方校招页 HTML → 廉价正则预筛 → LLM 抽网申日期」的抽取器，SSR 有数据的（字节等）一次覆盖；SPA 空壳（腾讯/百度）**诚实留白，不编**。拿不到就不显精确日期。
2. **批次差异只做「时间」差异**（提前批比正式批约早 N 周，cycle 级，来自官方时间线 / P2 据往年）；**HC 硬差异彻底砍**（数据不存在）；软经验难度差异**不做**（那是要避开的慢搜索路 + 重合规）。

### P3 定稿范围（据以上）

- **P3a 抽屉外露**（UI，数据无关，先做）：校招公司卡加「公司洞察 N」按钮（放「展开岗位」旁），点开现成 `CompanyInsightDrawer`（props = `{company, open, onClose}`，自取数）。复用 JobCard 现成的 `lib/insight-client` 微批拉取 + 「洞察 N / 岗位聚合 / 暂无」预告徽标，但提到**公司卡级只拉一次**（比现在每个 JobCard 各拉更省）；`暂无` 不显按钮（不给空抽屉）。
- **P3b 今年精确日期**：
  - **主力 = 快路②**：复用 P2 现成「抽取→判官→官方门→写」骨架，把数据源从慢搜索换成 httpx 并发直取官方校招页。流程：`sources` 取官方招聘域名 → 拼/探官方校招页 URL → httpx 并发 fetch → **廉价正则预筛**（无「网申/投递/X月X日」信号的 SPA 空壳直接跳过，不喂 LLM）→ `insight_engine.chat_json` 抽「批次 + 网申起止精确日期」→ `official_gate`（grounding host = 公司官方页 → auto-verify）→ 写 RCO `time_expr_type='精确日期'` + `date_start/date_end` + `source_kind='official_notice'` + `verify_status='verified'`。
  - **兜底 = 快路①**：对自有库有**干净近未来 deadline** 的公司（主要传统制造/央企），派生「今年约 X 截止」写 RCO `source_kind='derived_jobs'`。**必须清洗 text deadline**（排除 `长期有效`/`3000-01-01`/远未来占位/过去日期）。因批次标签不在岗位上，快路① 只能给**公司级**一个日期（不拆批次）。
  - **冲突时**：官方公告（快路②，`official_notice`）> 自有派生（快路①，`derived_jobs`）。展示措辞三档区分：`据官方公告` > `据在招岗位` > `据往年`。
  - **合规（§7-7 定）**：自有官方岗派生的 deadline **算「官方可复查证据」**（是从官方岗位页抓来的），可写 `verify_status='verified'`，但措辞用弱档「据在招岗位」，不冒充官方公告。
- **P3c 批次「时间」差异**（基本是 P3b 副产品）：P2 的 `campusTimelineSummary` 已显各批次 `batchBits`；P3b 把今年精确多批次日期填入后，顺势提炼一句「提前批比正式批约早 N 周」时间差 bit，显卡/抽屉。纯 cycle 级派生，**不碰 HC/难度**。
- **数据模型**：全部写现成 `recruitment_cycle_observations`（迁移 182 的 `date_start/date_end/time_expr_type` 已预留），**不新建表**；批次 HC 差异不落库（砍）；批次时间差读时派生。
- **提速工程**：快路② 并发 httpx（不吃搜索额度、分钟级）+ 快路① 纯读派生；通用搜索退为兜底；campus-cycle 主流程不再被搜索日额度卡死。

### 落地顺序（定）
P3a（抽屉外露，独立小改，先落见效）→ P3b 快路② 官方页抽取器（主力）→ P3b 快路① deadline 兜底（含清洗）→ P3c 时间差 bit（P3b 副产品）→ 提速工程收尾。

## 1. 背景与目标

- **产品**：求职雷达校招专区。P1=窗口徽章（此刻开没开，事实层）；P2=往年时间线（据往年·提前批约7月，据往年层）；P3=今年精确日期（官方源门）+ 批次深度差异洞察。

- **红线不变**：可靠度分层、诚实标来源与年份/时效、宁缺不编、精确日期只认官方可复查证据。

## 3. P3 范围（三块）

### A. 校招专区「公司洞察抽屉」外露（UI，创始人反馈）

- 现状：校招卡只有 公司名 + 窗口徽章 + 时间线一行 + 岗位计数 + 展开岗位列表；公司洞察（薪资/强度/文化/路径/时机）只能点进卡内某个 JobCard 才打开抽屉——**埋太深**。
- P3：在校招卡上加一个**直接入口**打开该公司的 `CompanyInsightDrawer`。形态待定（§7 未决项1）：一个「公司洞察」按钮 / 把最关键的洞察芯片（如薪资档、强度、时机）直接铺在卡面、点击展开抽屉看全部。
- 复用现成 `CompanyInsightDrawer`（portal 到 body），只是新增一个打开它的入口 + 传 company。

### B. 今年精确日期（③ 层，只认官方源）

- 目标：卡上/抽屉里显「2027届 · 正式批网申 8月15日–9月10日 截止」这种**今年确切日期**，而非只「据往年约8-9月」。
- 红线（继承设计）：`time_expr_type='精确日期'` **只接受官方校招公告 / 官方招聘页可复查证据**，拿不到绝不编。写 RCO 的 `date_start/date_end`。
- 展示上要和「据往年」明确区分（今年确切 = 更强的措辞 + 官方来源链接）。

### C. 提前批 vs 正式批 差异洞察（应届生信息差最大）

拆两层可靠度（**建议 P3 先做硬数据层，软经验层看 §7 未决项3**）：
- **硬数据层（可从自有岗位派生，高信任）**：提前批 vs 正式批的 **HC 体量差异**（各批次在招/历史岗位数）、**开放时间差**（提前批比正式批早几周）、**城市/职能覆盖差异**。这些能从我们已抓的校招岗直接算，属事实。
- **软经验层（经验型，多源共识，做浅重免责）**：难度/流程/竞争激烈度差异（「提前批 HC 少但竞争小、正式批 HC 多、补录门槛低」这类）。走 §4 快路③ + 多源共识门，标「据公开讨论」，不指名个人、去标识。

## 4. 爬虫提速策略（**本 spec 核心，创始人最强调**）

**诊断**：P2 Track B 慢，因为主源选错了——用「通用网络搜索」当主源，受搜索 API 日额度硬顶（~90-140 次/天且与 t3 共享），逐家 1-2 搜 → 实测 ~8 家/天。要覆盖塌陷行业几百家 + 精确日期，得几周，且反复重搜。**解法 = 换更快、额度不受限、且更权威的主源；通用搜索降级为兜底。** 三条快路，按「快 + 信任」排序：

### 快路① 从自有已抓岗位派生（零新爬取、即时、最高信任）——**首选**
- 我们**已经**每天抓这些公司的校招岗（`jobs` 表，含 `title`(届别/批次词)、`deadline`、`first_seen_at`、`job_type`、`city`、`education`）。这是**我们自己从官方岗位抓的数据，最快（读时即得）、最权威（官方岗）**。
- 派生什么：
  - **今年精确日期**：当前在招校招岗的 `deadline` 分布 → 「今年正式批网申约 X月X日截止」（这是今年真实数据，甚至比官方公告页更实时）。`first_seen_at` 分布 → 各批次今年实际开放时间。
  - **批次 HC 差异**：按 `title` 的批次词（提前批/正式批/补录/管培）聚合校招岗数 → 提前批 vs 正式批 HC 体量对比（硬事实）。
  - **当前批次窗口**：结合 P2 往年时间线 + 今年 first_seen/deadline，判「今年提前批已开/正式批未开」。
- 合规：这是自有官方岗派生，属**事实层**（同 Module B 的 T1 派生），可高置信展示。**接手 session 先 A0 live 验**：拉几家 seeded 公司的校招岗，看 title 能不能稳定分批次、deadline 覆盖率够不够（很多岗 deadline 为空是已知问题）。
- 工程：纯读 HK jobs 库 + 纯函数聚合，**零新爬虫、零搜索额度**，可全量几百家一次算完。**这条能独立支撑「今年精确日期 + HC 差异」大部分，且瞬时。**

### 快路② 官方校招页直取（httpx 并发、快、权威、不吃搜索额度）——**补充精确日期**
- 很多公司官网有**校招 landing / 校招时间轴页**（如 `jobs.bytedance.com/campus` 常有「网申时间 X月X日」）。我们在 `sources` 里已有这些公司的官方招聘域名。
- 做法：httpx 直接 fetch 官方校招页 HTML → LLM（或正则兜底常见「网申时间：…」模式）抽批次+精确日期 → 官方源 → 过官方源门 → 可 auto-verify（含 `time_expr_type='精确日期'`）。
- 为什么快：**不走搜索 API**（无日额度硬顶），只受 httpx 并发限制 → 几百页可并发在几分钟内抓完，不是几天。
- 复用：现有 adapter 框架 / `probe.py` 的 fetch + title-verify 工艺；抽取复用 `insight_engine.chat_json`。
- 难点（§7 未决项4）：每家校招页结构不同，通用抽取器怎么做（LLM 抽 vs 每家写规则）；哪些公司有可直取的校招公告页（先 A0 live 摸一批）。

### 快路③ 校招时间轴聚合页（一次抓多家、覆盖长尾）——**补广度 + 软经验**
- 应届生求职站 / 校招日历类页面发布「2027 秋招时间轴汇总」，**一页覆盖几十家**公司的批次时间。
- 一次 fetch → 多家时间 + 批次差异讨论。第三方 → 走**多源共识门**（选项 B，≥2 源重叠即发布，标「据公开信息」）。
- 用途：给官方页难抓的长尾公司补覆盖 + 批次难度/流程的软经验洞察（§3.C 软层）。

### 通用搜索（现 Track B）→ 降级为兜底
- 快路①②③ 都够不着的公司，才回退现有 `search_router` 通用搜索。不再当主源 → 不再被日额度卡死主流程。

### 提速工程（次要，但要做）
- **campus-cycle 搜索预算与 insight-enrich-t3 解耦**（现共用 `search_usage` 池、还错峰 45 分钟互抢）→ 各自独立日顶，或校招走独立 provider。
- 快路①②是并发/批量友好的（读库 + httpx 并发），把 cron 从「逐家串行 limit=8」改成「快路①全量派生 + 快路②并发直取一批」。
- 可选：配 `BOCHA_API_KEY` / 上调 `*_DAILY_CAP`（repo Variables）给兜底搜索加头寸。

**一句话**：**把主源从「慢的通用搜索」换成「快路①自有岗位派生（瞬时）+ 快路②官方页并发直取（分钟级）」，通用搜索退为兜底。** 这样 P3 才可能「快到有价值」——今年精确日期 + HC 差异能全量近实时出，而不是几周爬一遍。

## 5. 可靠度分层（P3 落到第 ③ 层 + 深度）

| 层 | 信息 | 主源（提速后） | 门 | 状态 |
|---|---|---|---|---|
| ① 事实·此刻 | 窗口开没开 | 抓取派生 | 直接展示 | ✅ P1 |
| ② 据往年 | 往年提前批/正式批月份 | 时间线（P2） | 官方门/多源共识 | ✅ P2 |
| ③ 今年精确 | 今年确切日期 | **快路①自有岗位 deadline + 快路②官方页** | **官方源门（严格）** | 🔜 P3 |
| ③+ 批次差异 | 提前批 vs 正式批 HC/难度 | 快路①（HC 硬数据）+ 快路③（软经验） | HC=事实层；难度=多源共识+免责 | 🔜 P3 |

## 6. 数据模型（尽量复用，别乱建表）

- **精确日期**：直接写 `recruitment_cycle_observations`，`time_expr_type='精确日期'` + `date_start/date_end`（字段迁移 182 已预留）+ `source_kind='official_notice'/'derived_jobs'` + `verify_status='verified'`（官方/自有派生）。P2 的读层/展示层已认这张表，卡片/抽屉小改即可显日期。
- **批次 HC 差异（硬数据）**：可**读时派生、不落库**（同 Module B T1 派生思路，从 jobs 现算），或落一个轻结构。建议先读时派生（最快、总是最新），§7 未决项5 定。
- **批次难度（软经验）**：若做，走 `insight_items`（experience grade + 免责 + 去标识 + 多源）或 RCO 的 payload——§7 未决项3 定要不要做、落哪。
- **不变量继承**：事实字段 immutable、据往年/精确日期都绑 `grad_class`、过期退役不删。

## 7. 未决项（接手 session 先跟创始人 brainstorm 清楚再动手）

1. **洞察抽屉外露的具体形态**：卡上加「公司洞察」按钮 → 打开抽屉？还是把最关键 2-3 个洞察芯片（薪资档/强度/时机）直接铺卡面、点开看全部？信息密度 vs 简洁怎么平衡（校招卡已经有徽章+时间线+岗位，别太挤）。
2. **今年精确日期：快路①（自有岗位 deadline 派生）vs 快路②（官方页直取）谁主谁辅、冲突怎么合**？自有岗位 deadline 是「今年真实」但覆盖看抓取质量（很多岗 deadline 空）；官方公告页是「官方声明」但要逐家抓。两者都有时以谁为准？
3. **批次差异做到哪**：P3 只做「HC 体量/开放时间差」硬数据（能从自有岗位派生、安全），还是也做「难度/流程/竞争」软经验（要多源+judge+免责、更重）？创始人上次（P2）已把「该先投哪批」的难度差异推到 P3——现在真做的话做多深。
4. **官方校招页通用抽取器**：每家校招页结构不同——统一 LLM 抽取（贵但通用）还是每家写规则（准但不 scale）？先 A0 live 摸一批 seeded 公司的官方校招页，看有多少家有可直取的「校招时间/网申日期」页、结构多杂。
5. **批次 HC 差异：读时派生 vs 落库**？读时最新但每次算；落库省算力但要维护新鲜度。
6. **提速优先级**：快路①②③ + 兜底，P3 先落哪几条？（建议：快路① 自有派生先落——最快见效、支撑精确日期+HC；快路② 官方直取次之补精确日期；快路③ 聚合页/软经验最后。）
7. **合规确认**：自有已抓官方岗派生出的「精确日期」算不算「官方可复查证据」（应该算——是我们从官方岗位页抓的）？这决定快路①能否直接 auto-verify 精确日期。

## 8. 非目标（P3 不做）

- 通知/推送（整个校招专区仍不做）。
- 自动投递 / 登录企业系统 / 绕验证码（硬边界不变）。
- 海外校招（先国内）。
- 逐个自然人的点名信息（去标识铁律）。
- 无官方证据的「今年精确日期」——拿不到绝不编（宁缺不编）。

## 9. 落地顺序建议（供接手 session 定）

- **A0（前置）**：live 摸底——① 拉几家 seeded 公司的校招岗，验「title 分批次 + deadline 覆盖率」够不够支撑快路①；② fetch 几家官方校招页，看有多少有可直取的精确日期、结构多杂（决定快路②可行性 + 抽取器形态）。**摸不通的快路诚实记录、本轮不做，不硬猜。**
- **P3a**：洞察抽屉外露（UI，独立小改，先落、见效快）。
- **P3b**：快路① 自有岗位派生（纯读+纯函数）→ 今年精确日期（deadline 派生）+ 批次 HC 差异 → 卡片/抽屉展示。**这条最快、最该先做。**
- **P3c**：快路② 官方校招页并发直取 → 补精确日期（官方源门 auto-verify）。
- **P3d**：快路③ 聚合页 + 软经验批次难度（多源共识，若 §7 决定做）。
- **P3e**：提速工程（campus-cycle 预算解耦 + 并发化）。
