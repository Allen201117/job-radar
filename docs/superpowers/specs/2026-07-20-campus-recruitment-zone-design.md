# 校招专区（Campus Recruitment Zone）设计

> 状态：设计定稿待评审 · 2026-07-20
> 作者：Claude（与创始人 + Codex 红队共创）
> 关联记忆：[[job-radar-must-apply-breakthrough]]（必投清单覆盖攻坚）

## 0. 一句话

给应届生一个专属阵地「校招专区」：按你的行业锁定必投清单那批目标公司，把每家**已接入官方校招源、且持续验证有效**的校招岗聚合到一张公司卡上，用**诚实的窗口徽章**告诉你哪家在招、哪家我们还没接入，接住此刻正在开的秋招（春招同一入口复用）。

## 1. 背景与目标

- **产品地基论**（创始人定调）：岗位库必须全面 / 可靠 / 健康 / 实时有效，地基打好才谈增长与商业化。当前热门行业只有零星几家几个岗，用户第一眼就不信任。
- **校招是最大用户群、且此刻正当时**：2026-07 中旬秋招提前批正在陆续开。产品对校招几乎没做专门支持——"秋招/春招/提前批/正式批"在代码里根本不是一等概念（`lib/china-keyword-expansion.js` 的分类函数完全不认这些词）。
- **实测供给缺口（2026-07-20 香港库）**：国内必投 330 家中 178 家有在招岗，其中 **131 家有校招岗**。分行业极不均：互联网 25、汽车 20、制造 15、消费 14、医疗 12、地产 12、能源 9、金融 8、教育 7、物流 6、传媒 3（均 /30）。约 47 家"有社招岗没校招岗"（大概率有校招板块只是没抓），152 家完全无岗。

**本设计目标**：把"校招求职者的核心焦虑"翻译成一个可靠、诚实、能接住提前批的产品阵地，并借此把"校招供给"这块地基补起来。

## 2. 用户与核心痛点（校招求职者视角）

按尖锐度：

1. **时间焦虑（最尖锐、信息差最大）**：错过一批可能等一年。用户要"打开就知道哪些窗口开着、哪些快截止"，不是翻进抽屉才看到。
2. **目标公司的确定性**：不缺岗位列表，缺"我这行该盯的几十家 + 它们的校招岗都在这、不用一个个官网翻"。
3. **批次信息差**：提前批 vs 正式批难度/流程差异、往年几月招满——老鸟知道、应届生不知道。
4. **链接可靠**：校招岗点开必须能投；死链在这个时间窗紧、试错成本高的群体里是致命的。

## 3. 已确认的设计决策（创始人拍板）

1. **目标公司 = 复用必投清单**（`lib/must-apply-list.ts`，每行业约 30 家），不单开名单；专区取用户行业对应的那批公司的**校招板块**岗位。
2. **时间信息分层可靠**：① 窗口状态（此刻开没开，纯抓取派生，全公司覆盖，最可靠）② 往年规律/黄金期/提前批 vs 正式批（头部人工策展 + LLM 辅助草稿必人工核对，标"据往年 + 具体年份"）③ 今年精确日期（只在有官方校招公告源时显示，拿不到绝不编）。
3. **独立一级入口「校招专区」**，秋招/春招共用（不叫"秋招雷达"），与"值得投""职业路径"平级。
4. **分期**：P1 = 目标公司聚合 + 诚实窗口徽章 + 定向补校招供给 + 可靠性三件套；P2 = 时间线洞察 + "校招批次"升为一等概念；P3 = 精确日期。
5. **通知/主动推送不做**（本期非目标）——先打地基，专区本身就是每天刷一眼的站内阵地。

## 4. P1 详细设计（本期实现范围）

### 4.1 数据来源与「校招准入门」

- **零新表、零 schema 改动**（分类走读时纯函数）。用户进专区 → 读其行业（`candidate_profiles` / `user_preferences`，复用 `resolveMustApplyIndustries`）→ 映射必投清单约 30 家 pattern → 一条 SQL 按公司聚合这些公司的在招岗（复用 `getMustApplyCoverage` 的单次全表扫描范式，避免 330 pattern × 26 万岗双 join 超时）。
- **专区专属「高置信校招准入门」**（比通用 `recruitmentCategory` 更严，精度优先）：新增纯函数 `lib/campus-zone.ts` 的 `campusAdmission(job) → 'campus' | 'intern' | 'pending' | 'reject'`：
  - **强准入 campus**：源 `job_type` 明确校招 / URL 门户校招路径（`/campus`、`/xiaozhao`）/ 标题含届别或校招项目词（20XX届、校园招聘、管培生、留学生专项、campus recruit、graduate program）。
  - **intern**：明确实习信号，单独成桶、可筛，**不默认混入校招**（暑期/日常/转正实习求职动作不同）。
  - **pending**：弱关键词命中但无强信号 → 进"待确认"，**不进默认列表**。
  - **reject**：≥2 年经验要求 / 明确社招 → 排除。
  - 原则：**先保证"展示的都真是校招"，再逐步补召回**。
- **质量金标准**：建立人工标注抽样集（按公司 / ATS 类型），持续测准入门的精确率与漏检率，作为回归基线。

### 4.2 窗口状态徽章（① 层，纯抓取派生，绝不 LLM/猜）

`lib/campus-zone.ts` 的 `windowStatus(companyAgg) → WindowState`。P1 只三个**诚实事实态** + 一个新鲜度降级态：

| 徽章 | 含义 | 判定 |
|---|---|---|
| 🟢 招聘中 | 有在招校招岗 | 该公司当前 ≥1 个通过准入门的校招岗，且来源新鲜度未过期 |
| ⚪ 当前未观测到在招校招岗 | 我们在盯、但此刻没看到（**不等于"没开"**） | 有该公司的校招源 / 历史，但当前 0 校招岗 |
| ⚙️ 待接入 | **诚实告知**：这家的校招源我们还没接 | 库里没这家校招源。子原因（运营可见，用户仅见"待接入"）：`no_source`（尚未接入）/ `source_only_social`（只接了社招板块）/ `crawl_error`（抓取异常） |
| ⏳ 数据待更新 | 新鲜度护栏降级 | 该公司源最后成功抓取超过阈值（如 > 72h）→ 不展示 🟢，降级此态 |

- **🔥"近期新增/提前批刚开" 砍到 P2**：first_seen 被 2026-06-15 库重建污染 + 列表重抓会刷新 → "库首次观察"≠"市场首次开放"。P1 不做突增判断，避免误报"提前批"。P2 有稳定基线（同源基线期 + 外部指纹 + 连续两次抓取都在 + 新接入/重建/URL 变更进冷却期）后再做，文案也只到"近期新增较多"，除非有官方批次证据不推断"提前批"。
- **季节感知属"解释层"不污染"事实层"**：徽章只陈述"我们此刻看没看到岗"；"按往年此刻应处黄金期/尾声"是 P2 洞察，卡上另起一行、明确来源，绝不改写事实态。
- **约 30 家全展示**（含 ⚪/⚙️）：应届生要的就是"我盯着这几家，谁开了我知道"。

### 4.3 校招专区页面（新一级路由 `/campus`）

- **形态**：仿 `/path` 建一个 force-dynamic 服务端页 + client 组件 + `loading.tsx` 骨架（遵循冷启动规范）。`components/Navbar.tsx` 加一级入口。
- **数据流**：SSR 读用户行业 → 公司聚合 → 每家一张公司卡。公司卡 = 公司名 + 窗口徽章 + 校招岗计数 + 展开区（该公司校招岗列表，复用 `JobCard`）。
- **排序（默认按"别错过"，不按岗位数）**：岗位排序 = 有明确截止日期的按截止升序在前（临近的置顶）→ 无截止日期的按新增时间降序在后；**明确区分"截止未知"与"持续招聘"**，不把"未抓到截止日期"暗示成"长期开放"。公司卡排序：🟢 招聘中优先、按"该公司岗位里最临近的已知截止"升序 → ⚪ → ⚙️。匹配度作为同档内次级排序键。
- **同公司多批次/事业群/城市归组**：公司卡内可解释分组（按批次/城市），避免 50 个雷同岗淹没关键信息。
- **筛选**：届别、学历、专业/职能、城市、"校招 / 实习"切换（intern 单独桶）。否则"适合我"的承诺不成立。
- **诚实文案**：不承诺"全部校招岗"，页面口径统一为"**已接入官方校招源并持续验证的岗位**"。

### 4.4 可靠性三件套（P1 必带，这是"可靠"的兑现）

1. **来源新鲜度**：公司源最后成功抓取时间过阈值 → 徽章降级 ⏳，不拿旧数据冒充在招。
2. **岗位失效下架**：复用现有失活治理（sweep + 展示时校验 `lib/liveness-client.js`）——专区展示的校招岗加载后异步探活，死岗当场隐藏。跳转到社招 / 岗位 ID 复用也要能识别下架。
3. **外跳链接抽检**：定期抽检专区岗位的 `jd_url` 仍为可投的官网详情页（复用 enrich/liveness 信号），失效进下架队列。

### 4.5 定向补校招供给（P1 的地基活，非新代码、是数据/运营活）

- **审计**：盘必投各公司校招板块覆盖，产出三类清单——`no_source`（152 家无岗里真无源的）/ `source_only_social`（47 家有社招没校招的）/ `crawl_error`。
- **补源**：给 `source_only_social` 家补校招板块 URL（beisen `/campus`、moka `campus_apply`、feishu 校招板块等），**完全复用上一轮必投攻坚的 discover_domestic + 浏览器 confirm + 探活入库工艺**（[[job-radar-must-apply-breakthrough]]）。
- **补源优先级（Codex 修正：不只看行业塌陷）**：`公司对目标用户重要性 × 预期校招岗体量 × ATS 可接入性 × 当前信息缺口`。塌陷行业（传媒 3 / 物流 6 / 教育 7 / 金融 8）作为其中"信息缺口"维度的权重，但不单独定序——别在难抓又低价值的公司上耗时间。
- **可见性**：每补一家，其徽章从 ⚙️ → 🟢/⚪，用户直接看到专区长肉；⚙️ 的子原因让运营能追"这是没源还是抓炸了"。

### 4.6 用户纠错入口

- 岗位/公司卡上一个轻量入口："这不是校招 / 链接失效 / 已结束" → 进复核队列（复用现有洞察申诉 `insight_disputes` 那套的模式：用户可插、admin 处理）。反馈直接改善准入门与失活判定。

## 5. 组件边界与接口

- `lib/campus-zone.ts`（纯函数，可 node --test）：`campusAdmission(job)`、`windowStatus(companyAgg)`、公司卡排序比较器、多批次归组。**无 LLM、无网络、无 DB**——纯输入输出，独立可测。
- `lib/jobs-store/read.ts` 新增 `getCampusZone(industryPatterns) → CompanyCampusAgg[]`：单次全表扫描按公司聚合校招岗 + 新鲜度 + first_seen 分布 + 岗位样本。
- `app/campus/page.tsx`（SSR）+ `app/campus/campus-client.tsx` + `app/campus/loading.tsx`。
- `app/api/campus-zone/*`（如需异步补数）；纠错走 `app/api/campus-zone/dispute` 或复用洞察申诉端点。
- 复用：`lib/must-apply-list.ts`（行业→公司）、`JobCard`、`lib/liveness-client.js`（展示时校验）、`Navbar`（入口）。

## 6. 错误处理

- 用户无行业偏好 → 兜底"互联网/科技"（`resolveMustApplyIndustries` 已有此兜底），并提示"完善简历行业以精准锁定目标公司"。
- 某公司聚合查询失败 → 该卡显示 ⏳ 数据待更新，不拖垮整页（各卡独立）。
- 校招岗为 0 的行业（如传媒当前仅 3 家有岗）→ 页面**不隐藏空卡**，用 ⚙️/⚪ 诚实呈现，并展示补源进度感（"我们正在接入 X 家的校招源"）。

## 7. 测试

- **纯函数优先**：`tests/campus-zone.test.js` 覆盖 `campusAdmission`（强/弱/实习/社招/边界）、`windowStatus`（三态 + 新鲜度降级 + 子原因）、排序、归组。
- **金标准回归**：准入门精确率/漏检率抽样集（人工标注），上线前保精度。
- crawler 补源走现有 `crawler/test_*.py` 风格 unittest（不打真实网络）。
- 不改现有 `recruitmentCategory` 的话，其四套现有测试不受影响；若微调需同步。

## 8. P2 / P3 展望（本期不实现，记录形状）

- **P2 招聘周期观测表** `recruitment_cycle_observations`（新表，可版本化，洞察层引用它、不塞进 `insight_items`）：
  - 维度：公司、招聘年度 + 毕业届别、批次（提前/正式/补录）、事件（开放/截止/黄金期）、时间表达类型（精确日期/范围/月/历史规律）、起止值、置信度、证据 URL、证据抓取时间、人工核验状态、失效时间。
  - **"据往年"必须绑定具体年份**（2024 规律不能说成泛"往年"；区分"2027 届招聘"与"2026 年发生的招聘"）。
  - **原始 observation 不可覆盖**，展示结论从中派生——防 LLM/编辑误改无法追溯。
  - LLM 只提取候选 + 标注证据片段；无官方原文 / 无法定位公司+届别 / 日期冲突 → 禁止发布为日期事实。精确日期只接受官方校招公告或官方招聘页可复查证据。
  - "校招/秋招/春招/提前批/正式批"升为一等概念（分类函数认这些词 + 结构化批次字段）。
  - 🔥"近期新增较多"徽章（有稳定基线后）。
- **P3**：今年精确日期（③ 层，官方源门）；提前批 vs 正式批难度/流程差异等深度信息差洞察。

## 9. 非目标（本期明确不做）

- 通知 / 主动推送（邮件/站内提醒）——先打地基。
- 自动投递、绕验证码、登录企业系统（Phase 1 硬边界不变）。
- 海外校招（本期专区先做国内必投；海外必投清单结构已就绪，后续可复用同套页面）。
- 春招独立逻辑（同入口复用，季节差异由 P2 时间线承载，P1 不为春招单独建代码）。
