# 求职雷达 PRD v2 — 岗位检索 + 职业洞察（双层产品方案）

> **文档性质**：以 PRD（产品需求文档）为骨架，融入商业模式 / 核心价值（BP 内容）的完整方案。
> **目标读者**：实现本产品的开发 Agent、工程师、设计、以及合伙人 / 投资人。
> **版本**：v2（2026-06-02）。本文件取代并扩展 `1780305970246_PRD.md`（下称「v1 PRD」），向后兼容 v1 的所有数据结构与质量门。
> **一句话定位**：面向中国求职场景，把「**真实可投递的官方岗位**」与「**只有资深从业者才知道的职业信息差**」组合成一个产品，让用户不仅看到「有什么岗位」，更知道「值不值得投、什么时候投、怎么投得进」。

## 0. 如何阅读本文档（给 Agent 的导航）

本产品由**两个数据层**和**四个功能模块**组成。实现时请严格区分两层，**永远不要让它们的数据互相污染**：

| 层 | 性质 | 信任来源 | 对应模块 |
|---|---|---|---|
| **岗位数据层** | 官方、确定、可点击 | 官方招聘源 + 严格质量门 | 模块 A（岗位检索闭环）、模块 C（简历匹配）、模块 D（看板） |
| **职业洞察层** | 聚合、归因、分级标注 | 公开信息 + 社区观点（去标识化） | 模块 B（职业洞察） |

核心实现顺序见 §13 MVP 与路线图。数据模型见 §11。失败必须可解释，见 §12。**边界（不做什么）见 §14，违反边界等同于实现错误。**

## 1. 产品定位与愿景

### 1.1 定位
面向中国求职场景的**官方岗位雷达 + 职业决策伙伴**。

覆盖范围必须广，不应只局限于少数固定公司：
- 中国公司（互联网大厂、硬科技、新能源、制造业、银行券商、央国企）
- 在华外企
- 外企的中国岗位（全球官网中的 China 岗位）
- 科研 / 实验室机构

### 1.2 愿景
单纯的岗位聚合是工具，护城河浅、用户找到工作就走、粘性低。本产品的长期价值在于**职业洞察层**——把散落在脉脉、小红书、即刻、知乎、一亩三分地等处、通常只有从业十余年的资深人士才掌握的「信息差」，结构化、个性化地交付给正在做决策的求职者。

**愿景一句话**：让求职者像有一个「在目标公司干了十年的朋友」在旁边给建议。

---

## 2. 核心价值主张（为什么是「岗位检索 + 职业洞察」）

### 2.1 两个模块的关系
- **岗位检索是获客入口**：高频、刚需、容易理解，负责把用户拉进来。
- **职业洞察是核心与付费点**：难复制、用户愿付费、会反复回来用，负责留住用户并变现。

二者互补：
- 岗位数据层告诉用户「**有哪些坑位**」；
- 职业洞察层告诉用户「**这个坑值不值得跳、什么时候跳、怎么跳进去**」。

### 2.2 职业洞察层解决的真实痛点（举例）
这些是真实存在、却没有产品系统化交付的「圈内知识」：
- **时机类**：微软中国受财年影响，每年约 5–7 月通常没有 headcount，此时官网即便挂岗，投递也难有进展；不同公司财年截止日不同（有的 6 月底、有的年底），HC 节奏随之不同。
- **路径类**：想进字节，常见路径是先去快手 / 小红书积累履历，这三家互相青睐对方员工。
- **性价比类**：中国互联网里，字节给薪偏高、拼多多强度偏大、京东门槛相对低。
- **文化 / 避坑类**：加班文化、管理风格、业务稳定性等。

> ⚠️ 上述例子在产品中**必须以「聚合 + 归因 + 时间戳」形式呈现**（见 §7.2、§8），不得作为产品断言输出。

## 3. 商业模式（BP 部分）

### 3.1 价值与护城河
- 岗位聚合谁都能做（招聘平台、官网、爬虫脚本），**护城河浅**。
- 真正的壁垒是**职业洞察层的「信息分级 + 验证 + 负责任表达」机制**——不是「能爬到信息」，而是「能辨别真伪并敢于结构化交付」。爬取人人会，辨别和担责才难。

### 3.2 收费设想（待验证，非 MVP 必需）
- **免费层**：岗位检索 + 看板 + 基础匹配。负责获客与留存。
- **付费层**：职业洞察层的深度内容（公司求职日历、薪资带、跳槽路径、避坑报告）、个性化职业路径规划。
- 变现优先级低于「先验证洞察层是否有人买单」，详见 §13。

### 3.3 竞品与差异
市面已有公司点评 / 薪资分享类产品（看准网、职友集、各类 offer 薪资平台、脉脉公司点评），说明「聚合公司信息成产品」可行且有先例。**差异点**：本产品把洞察与**实时官方岗位 + 简历画像**打通，形成「发现岗位 → 判断岗位 → 投对岗位」的决策闭环，而非孤立的点评库。

---

## 4. 用户画像与核心场景

### 4.1 目标用户
正在求职 / 看机会的中国求职者：应届校招生、社招跳槽者、关注外企中国岗位的人群。

### 4.2 核心场景（User Story）
1. *作为求职者*，我输入关键词 / 城市 / 岗位类型，希望看到**真实、可点击投递的官方岗位**。
2. *作为求职者*，我看到一个岗位时，希望知道**这家公司现在是不是招聘窗口期、这个岗位的薪资强度、进这家公司的常见路径、有哪些坑**。
3. *作为求职者*，我粘贴 / 上传简历后，希望系统**按我的画像给岗位排序并打分**。
4. *作为求职者*，我希望对岗位标记 `saved / ignored / applied`，且这些状态**只属于我**。

## 6. 模块 A：岗位检索闭环（岗位数据层）

> 继承 v1 PRD 全部内容，此处为权威版本。核心原则：**官方、真实、可点击**，对来源保持「洁癖」。

### 6.1 分层检索（按需升级）
用户输入关键词 / 城市 / 岗位类型后，分层、按需升级，**而非每次全量调用**：
1. **本地 jobs 岗位库检索** — 先查已有岗位库。
2. **已确认官方源刷新** — 结果不足时，刷新已验证官方招聘源（如百度、京东、Apple、Siemens 等）。
3. **低频动态官方源发现** — 仍不足时，低频调用 search provider（如百度千帆），发现**新的官方招聘入口**：公司官网招聘页 / 校招官网 / 社招官网 / 外企全球官网的 China 岗位页 / 官方 ATS board / 具体岗位详情页。

### 6.2 来源过滤与沉淀
**必须过滤**第三方招聘平台与低质量来源：BOSS、猎聘、智联、前程无忧、LinkedIn、Indeed、知乎、公众号转载、SEO 聚合页、高校就业转载页。

> 注意：此处被过滤的来源（如知乎、公众号）在**岗位数据层**被拒绝；但其中的**观点性内容**可作为**职业洞察层**的素材进入模块 B，二者通道不同、处理方式不同（见 §8）。

- 被识别为**高质量官方招聘入口**的结果 → 沉淀到 `source_candidates`，供审核 / 解析 / 转化。
- 有 parser / adapter 支持的源 → 解析为真实 `jobs`。

### 6.3 jobs 写入质量门（严格，不可放宽）
写入 `jobs` 必须**全部满足**：
- `company`、`title`、`jd_url` 非空；
- `jd_url` 是**用户可点击的官方岗位详情页**，不能是首页 / 列表页 / 搜索页 / API JSON；
- 页面 HTTP 200，且包含岗位 title 或核心标题片段；
- 同一 `jd_url` 不重复插入（唯一约束）。

### 6.4 覆盖扩展（持续扩大公司与 source）
不应停留在百度、京东、SHLAB 等少数源，应持续通过 `source_candidates` + known sources refresh + parser conversion 扩大可用岗位来源。
- **中国公司**：字节、腾讯、阿里、美团、华为、比亚迪、宁德时代等。
- **外企（在华 / 全球官网 China 岗位）**：Apple、Microsoft、Amazon、Siemens、Bosch、Tesla、J&J、AbbVie、Merck、Roche、Novartis 等。
- **招聘系统（ATS）**：Moka、北森、飞书招聘、Workday、SuccessFactors 等。

### 6.5 受控实时检索策略（如百度千帆）
定位 = **官方招聘源发现工具**，不是逐岗位搜索器。
- 默认一次用户点击只调用 **1 个** generated query；点击「继续发现更多」才调用第 2 个；
- 相同 query 在缓存时间内复用结果；
- 遇 429 进入 cooldown；
- 每次调用记录 `discovery_runs`，并沉淀 `source_candidates` / `jobs`；
- 无结果必须给**可解释 `failure_reason`**（见 §12）。

## 7. 合规与风险控制（关键约束，先于洞察层实现）

> 本节是**硬约束**。核心结论：**风险不来自「信息是内幕」，而来自「把不实信息当事实说」和「用不合规手段拿数据」。** 前者靠表达方式解决，后者靠数据来源选择解决。

### 7.1 误区澄清
我们要做的洞察，绝大部分**不是法律意义上的「商业秘密」**（商业秘密需满足：不为公众所知、有商业价值、权利人采取保密措施）。「字节给薪偏高」「微软财年期 HC 少」属于公开讨论、公司未保密的观察归纳，不构成「泄露内幕」。

### 7.2 三类真实风险及对应规避

| 风险 | 来源 | 规避原则（产品化为硬规则） |
|---|---|---|
| **名誉权 / 商誉** | 用事实口气说出可能不实的负面评价 | 永远「**聚合 + 归因**」：输出「据 N 位用户反馈…」「根据公开数据…」，**禁止**产品断言「我们认定 X 公司最累」。所有评价性洞察必须带样本量与来源。 |
| **数据爬取合规** | 抓取脉脉 / 小红书 / 知乎等（已有不正当竞争判例） | 优先官方 API、公开网页、有授权的数据源；尊重 robots 协议；**不绕过反爬技术措施**。此项为最高合规优先级，落地前须法律确认。 |
| **个人信息（PIPL）** | 帖子含具体个人薪资 / 姓名 | 只做**去标识化的群体性归纳**，`insight_items` 不得指向或可还原到具体自然人。 |
| **著作权** | 整段转载 UGC 原文 | 只提取事实与归纳观点，**不存储 / 不展示原文整段**；`insight_sources` 仅存链接、时间、必要摘要片段。 |

### 7.3 通知—删除机制（降低担责）
提供**申诉 / 删除通道**：公司或个人可对某条 `insight_item` 提出异议，系统能快速下架。此机制显著降低被诉时的责任。

### 7.4 表达分级（同时是壁垒与护身符）
所有洞察按 `fact` / `experience` / `rumor` 三级处理（见 §8.2）：事实可陈述（带来源），观点须标为观点，纯传闻要么不发要么重度免责。**这套分级既是法律护身符，也是产品的核心壁垒。**

## 8. 模块 B：职业洞察层（新增核心）

> 本模块是产品差异化的核心。实现前必须先落实 §7 的合规约束。

### 8.1 洞察维度（结构化为固定四类）
所有洞察归入以下 `dimension`，便于 AI 整理、用户理解、验证与更新：

| dimension | 含义 | 风险等级 | MVP 优先级 |
|---|---|---|---|
| `timing` 时机类 | HC 节奏、财年截止、校招 / 社招季、招聘窗口期 | **最低**（多可由公开信息推出，属事实陈述） | **最高，先做** |
| `compensation_intensity` 性价比类 | 薪资带、工作强度、入职门槛 | 中 | 中 |
| `path` 路径类 | 跳槽链路、对口公司、内推偏好 | 中 | 中 |
| `culture` 文化 / 避坑类 | 加班、管理风格、业务稳定性 | **较高**（负面评价居多） | **谨慎、做浅** |

> 原则：`timing` 做深做准，`culture` 做浅做克制。

### 8.2 信息分级与验证机制（产品壁垒）
每条 `insight_item` 必须带 `grade`：

| grade | 定义 | 处理方式 |
|---|---|---|
| `fact` 事实 | 可由公开来源验证（财报、官方校招公告、官网） | 可陈述，**必须带来源链接** |
| `experience` 经验 | 多位用户一致反馈的主观经验 | 标为「据 N 位用户反馈」，**须 ≥ 阈值样本量（如 N≥5）多源交叉印证后才输出** |
| `rumor` 传闻 | 单一 / 少量来源、未经印证 | **默认不输出**；如输出须重度免责且不进入付费内容 |

**验证流水线**（实现要点）：
```
原始素材 (insight_sources)
   → 去标识化 (移除可还原到个人的信息)
   → 归类 dimension
   → 多源交叉比对 → 定 grade
   → grade=fact:       校验公开来源链接有效
     grade=experience: 校验样本量 ≥ 阈值且来源多样
     grade=rumor:      拦截或打重度免责标记
   → 写入 insight_items (带 time_window 时效, sample_size, source_refs)
```

### 8.3 时效性（强制字段）
HC 节奏、组织架构会随财年 / 业务调整变化。每条 `insight_item` 必须带：
- `valid_from` / `valid_until` 或 `time_window`（如「每年 5–7 月」「FY2026」）；
- `last_verified_at`：上次核实时间；
- 过期洞察必须降权或标「可能已过时」，不得当作当前事实展示。

### 8.4 输出形态
- **公司画像卡（company_profile）**：在岗位详情 / 公司维度聚合展示四类洞察。
- **岗位旁洞察提示**：在岗位看板（模块 D）中，针对该公司当前的 `timing` 洞察给出提示（如「该公司当前可能处于非招聘窗口期，依据公开财年信息」）。
- **（路线图后期）个性化职业路径规划**：结合简历画像（模块 C）给出方向 / 路径 / 避坑建议。

### 8.5 数据来源
- 优先：公开网页、官方财报 / 校招公告、有授权或开放的数据源。
- 社区观点：仅做**去标识化归纳**，遵守 §7.2 的爬取合规约束，不存原文整段。

## 9. 模块 C：简历画像驱动匹配（岗位数据层）

> 继承 v1 PRD，边界不变。

- 用户粘贴简历文本，或上传 `.txt` / `.md`。
- **规则解析**（非 LLM）：目标岗位、目标城市、技能、行业、求职阶段、教育 / 经历摘要。
- 用户确认后同步到 `user_preferences`，用于岗位匹配评分与排序。
- **边界**：简历功能只服务岗位匹配，**不**扩展为简历优化器，**不**接 LLM，**不**做自动投递。

---

## 10. 模块 D：前端岗位看板（含洞察）

- **展示**：公司、岗位标题、城市、岗位类型、匹配分、岗位详情链接。
- **洞察集成**：岗位卡片 / 公司维度展示来自模块 B 的洞察卡片（带 grade 标签、来源、时间），与官方岗位信息**视觉上明确区分**。
- **用户操作**：`saved` / `ignored` / `applied`，**按用户隔离**（数据归属单一用户）。

## 11. 数据模型（给 Agent 的实现契约）

> 以下为逻辑 schema，字段命名向后兼容 v1。`*` 表示新增表 / 字段。

### `jobs`（岗位数据层，严格质量门）
| 字段 | 说明 |
|---|---|
| id | 主键 |
| company | 非空 |
| title | 非空 |
| jd_url | 非空、唯一、官方可点击详情页 |
| city | 城市 |
| job_type | 岗位类型 |
| http_status | 写入时须 200 |
| source_id | 关联来源 |
| created_at / updated_at | 时间戳 |

### `source_candidates`（候选官方招聘入口）
| 字段 | 说明 |

## 12. 失败模式与可解释性

任何一次检索 / 洞察请求**无结果时必须返回可解释的 `failure_reason`**，不得静默失败：
- 岗位层：`provider_rate_limited`、`all_results_rejected`、`candidates_pending`、`parser_missing`、`quality_gate_failed`。
- 洞察层（新增）：`insight_unverified`（样本不足或未交叉印证）、`insight_outdated`（仅有过期洞察）。

前端须把 `failure_reason` 翻译成用户可懂的提示（如「该公司暂无经核实的招聘节奏信息」）。

## 13. MVP 范围与路线图

### 阶段 0（已有 / v1 基础）
模块 A 岗位检索闭环 + 模块 C 简历匹配 + 模块 D 看板（基础版）。

### 阶段 1（MVP — 验证洞察层是否有人买单）
**不要一上来全公司全维度。**
- 选 **10–20 家高频大厂**；
- 只做最干净的 **`timing` 时机维度**（求职日历 / HC 节奏 / 校招社招季），辅以少量 `compensation_intensity`；
- 把这 10–20 家**做深做准**——做准 20 家远胜做浅 200 家，因为洞察一旦出错，信任即崩；
- 跑通验证流水线（§8.2）与合规约束（§7）。

### 阶段 2（打通闭环）
- 扩展 `path` 维度（跳槽链路）；
- 把洞察与简历画像（模块 C）结合，输出**个性化职业路径规划**雏形；
- 扩大公司覆盖。

### 阶段 3（变现）
- 免费层（检索 + 看板 + 基础匹配）/ 付费层（深度洞察 + 路径规划）分层；
- `culture` 维度谨慎引入。

## 14. 不做什么（边界，违反等同实现错误）

- **不**让洞察层（聚合 / 主观）污染岗位数据层（官方 / 确定）；二者数据通道、信任级别、UI 呈现必须分离。
- **不**以产品口吻断言对公司的负面评价；一切评价性内容必须聚合 + 归因 + 带样本与时间。
- **不**绕过目标站点的反爬技术措施；**不**存储 / 展示 UGC 整段原文。
- `insight_item` **不**指向或可还原到具体自然人（PIPL）。
- 简历模块**不**做优化器、**不**接 LLM、**不**自动投递（继承 v1 边界）。
- 岗位 `jd_url` **不**接受首页 / 列表页 / 搜索页 / API JSON（继承 v1 质量门）。

## 附：核心价值一句话回顾

> **岗位检索把用户引进来（工具、获客）；职业洞察把用户留下来并变现（认知服务、护城河）。**
> 真正的壁垒不是「能爬到信息」，而是「**信息分级 + 验证 + 负责任表达**」——这套机制同时是产品的差异化核心与法律护身符。
