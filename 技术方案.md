# Job Radar Private Beta 技术方案

版本：v0.1  
日期：2026-05-18  
阶段：3–5 人轻量内测版  
技术定位：Vercel + Supabase + GitHub Actions 的低成本岗位雷达看板  
成本目标：Phase 1 控制在 0–100 元/月

---

## 0. 文档目的

本文档用于指导开发 Agent 实现 **Job Radar Private Beta v0.1**。

当前目标是开发一个给 3–5 个早期用户使用的低成本官网岗位雷达看板。

本阶段不是商业化 SaaS，不需要重型架构。

明确不使用：

1. GPU；
2. 自托管 LLM；
3. K8s；
4. Redis；
5. Celery；
6. Prometheus/Grafana/Loki；
7. Elasticsearch；
8. 复杂多租户权限；
9. 付费订阅系统。

当前重点是跑通：

> 企业官网 → 每日抓取 → Supabase 岗位库 → 用户偏好排序 → Vercel 看板 → 官网跳转 → 用户操作反馈

## 2. 核心设计原则

### 2.1 低成本

Phase 1 尽量使用免费额度：

1. Vercel 免费档；
2. Supabase 免费档；
3. GitHub Actions 免费额度；
4. Phase 1 不用 LLM；
5. Phase 1 不做邮件推送。

### 2.2 单库优先

所有数据先放 Supabase Postgres。

不引入：

1. Redis；
2. Elasticsearch；
3. 独立向量数据库；
4. 数据仓库。

### 2.3 简单可靠

第一阶段优先保证：

1. 抓取能跑；
2. 数据能存；
3. 用户能看；
4. 操作能记录；
5. 不重复插入；
6. 不越过合规边界。

### 2.4 用户级隔离

所有用户共享岗位库，但用户偏好和操作必须按 user_id 隔离。

必须使用 Supabase RLS，防止用户看到或修改其他人的偏好与操作。

### 2.5 不绕过限制

抓取逻辑必须遵守：

1. 不登录；
2. 不绕验证码；
3. 不访问登录后页面；
4. 不破解接口；
5. 不使用代理池；
6. robots 明确禁止时停止；
7. 403/验证码/登录页时停止该源并记录。

## 4.1 profiles

用于扩展 Supabase Auth 用户。

```sql
create table profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  display_name text,
  role text default 'user',
  created_at timestamptz default now()
);
```

role 可选：

1. admin；
2. user。

---

## 4.2 sources

企业招聘源。

```sql
create table sources (
  id uuid primary key default gen_random_uuid(),
  company text not null,
  source_url text not null,
  source_type text,
  adapter_name text,
  crawl_method text default 'http',
  enabled boolean default true,
  last_checked_at timestamptz,
  notes text,
  created_at timestamptz default now()
);
```

字段说明：

- company：公司名，例如 Apple、百度；
- source_url：招聘页面入口；
- source_type：campus / social / internship / mixed；
- adapter_name：对应 crawler adapter；
- crawl_method：http / playwright / manual；
- enabled：是否启用；
- last_checked_at：最近抓取时间；
- notes：备注。

---

## 4.3 jobs

共享岗位库。

```sql
create table jobs (
  id uuid primary key default gen_random_uuid(),
  source_id uuid references sources(id),
  company text not null,
  title text not null,
  location text,
  job_type text,
  summary text,
  jd_url text not null,
  apply_url text,
  salary_text text,
  posted_at timestamptz,
  first_seen_at timestamptz default now(),
  last_seen_at timestamptz default now(),
  status text default 'active',
  content_hash text,
  created_at timestamptz default now(),
  unique(company, title, location, jd_url)
);
```

字段说明：

- title：岗位名；
- location：城市或地点；
- job_type：校招/社招/实习等；
- summary：简短描述；
- jd_url：岗位详情页；
- apply_url：投递链接，如果没有则为空；
- salary_text：官网披露的薪资文本；
- posted_at：官网发布时间，如果有；
- first_seen_at：系统首次发现时间；
- last_seen_at：系统最近确认时间；
- status：active / removed / expired / error；
- content_hash：岗位内容 hash，用于判断是否变化。

## 4.4 user_preferences

每个用户自己的求职偏好。

```sql
create table user_preferences (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references auth.users(id) on delete cascade,
  target_locations text[],
  target_roles text[],
  target_keywords text[],
  exclude_keywords text[],
  target_companies text[],
  daily_limit integer default 20,
  created_at timestamptz default now(),
  updated_at timestamptz default now(),
  unique(user_id)
);
```

字段说明：

- target_locations：目标城市；
- target_roles：目标岗位方向；
- target_keywords：希望命中的关键词；
- exclude_keywords：排除关键词；
- target_companies：关注公司；
- daily_limit：每日默认展示数量。

## 5. Row Level Security 设计

必须启用 RLS。

## 5.1 profiles

用户只能读写自己的 profile。

```sql
alter table profiles enable row level security;

create policy "Users can read own profile"
on profiles
for select
using (auth.uid() = id);

create policy "Users can update own profile"
on profiles
for update
using (auth.uid() = id)
with check (auth.uid() = id);
```

## 5.2 jobs

所有登录用户都可以读取 jobs。

```sql
alter table jobs enable row level security;

create policy "Authenticated users can read jobs"
on jobs
for select
using (auth.role() = 'authenticated');
```

前端不允许用户直接写 jobs。jobs 由 crawler 使用 service role 写入。

## 5.3 user_preferences

用户只能管理自己的偏好。

```sql
alter table user_preferences enable row level security;

create policy "Users can read own preferences"
on user_preferences
for select
using (auth.uid() = user_id);

create policy "Users can insert own preferences"
on user_preferences
for insert
with check (auth.uid() = user_id);

create policy "Users can update own preferences"
on user_preferences
for update
using (auth.uid() = user_id)
with check (auth.uid() = user_id);
```

## 5.4 job_actions

用户只能管理自己的岗位操作。

```sql
alter table job_actions enable row level security;

create policy "Users can read own actions"
on job_actions
for select
using (auth.uid() = user_id);

create policy "Users can insert own actions"
on job_actions
for insert
with check (auth.uid() = user_id);

create policy "Users can update own actions"
on job_actions
for update
using (auth.uid() = user_id)
with check (auth.uid() = user_id);

create policy "Users can delete own actions"
on job_actions
for delete
using (auth.uid() = user_id);
```

## 5.5 sources 与 crawl_runs

Phase 1 可以采用：

- 所有登录用户可读 sources；
- 只有 service role 或管理员可写 sources；
- crawl_runs 仅管理员可读。

前端不要暴露 service role key。

## 8. 规则匹配算法

文件：`apps/web/lib/scoring.ts`

输入：

1. job；
2. user_preferences；
3. user_actions。

输出：

1. score；
2. matched_keywords；
3. hidden_reason。

初始规则：

```text
score = 0

if title 命中 target_roles:
  score += 30

if location 命中 target_locations:
  score += 20

if company 命中 target_companies:
  score += 15

for keyword in target_keywords:
  if keyword 出现在 title 或 summary:
    score += 5

if first_seen_at 在 7 天内:
  score += 10

for keyword in exclude_keywords:
  if keyword 出现在 title 或 summary:
    score -= 50

if user_action 包含 ignored:
  hidden_reason = "ignored"

if user_action 包含 applied:
  hidden_reason = "applied_by_default"
```

展示逻辑：

1. ignored 默认不展示；
2. applied 默认不展示；
3. saved 可以展示并加标签；
4. 用户可以在筛选器中显示 ignored/applied。

## 9. Crawler 设计

## 9.1 运行方式

Crawler 由 GitHub Actions 每天触发。

也可以手动触发。

Crawler 使用 Supabase service role key 写入数据库。

注意：

- service role key 只能放在 GitHub Secrets；
- 不能写入前端；
- 不能提交到 Git。

## 9.2 抓取流程

文件：`crawler/run.py`

流程：

```text
1. 连接 Supabase
2. 读取 enabled sources
3. 对每个 source：
   a. 创建 crawl_runs 记录
   b. 检查 robots
   c. 根据 adapter_name 调用对应 adapter
   d. 获取岗位列表
   e. 标准化岗位
   f. 生成 content_hash
   g. upsert jobs
   h. 更新 sources.last_checked_at
   i. 更新 crawl_runs 结果
4. 输出运行总结
```

## 9.3 Adapter 接口

文件：`crawler/adapters/base.py`

```python
from dataclasses import dataclass
from typing import Optional, List

@dataclass
class RawJob:
    company: str
    title: str
    location: Optional[str]
    job_type: Optional[str]
    summary: Optional[str]
    jd_url: str
    apply_url: Optional[str] = None
    salary_text: Optional[str] = None
    posted_at: Optional[str] = None

class BaseAdapter:
    def fetch(self, source_url: str) -> str:
        raise NotImplementedError

    def parse(self, html: str) -> List[RawJob]:
        raise NotImplementedError
```

每个 source adapter 继承 BaseAdapter。

第一批 adapter：

1. baidu.py；
2. jd.py；
3. haier.py；
4. apple.py；
5. siemens.py。

如果某源难度太高，可以先标记 manual，不阻塞其他源。

## 9.4 标准化 normalizer

文件：`crawler/normalizer.py`

功能：

1. 清洗 title；
2. 清洗 location；
3. 标准化 URL；
4. 截断 summary 到 200–300 字；
5. 生成 content_hash；
6. 处理 salary_text；
7. 处理 posted_at。

content_hash 可以使用：

```python
import hashlib

def make_hash(title, location, summary):
    raw = f"{title}|{location}|{summary or ''}"
    return hashlib.sha256(raw.encode("utf-8")).hexdigest()
```

## 9.5 robots 检查

文件：`crawler/robots.py`

基础要求：

1. 对每个 source_url 检查 robots.txt；
2. 如果明确 disallow，则跳过该 source；
3. 如果 robots 无法读取，记录 warning，但不一定阻塞；
4. 如果出现验证码、登录、403，立即停止该 source。

robots 状态可以先写入 crawl_runs.error_message。

后续再加字段到 sources。

## 10. GitHub Actions 配置

文件：`.github/workflows/daily-crawl.yml`

```yaml
name: daily-job-crawl

on:
  schedule:
    - cron: '0 0 * * *' # UTC 00:00 = Singapore/China 08:00

        env:

```

## 12. LLM 使用策略

## Phase 1

不使用 LLM。

目的：

1. 降低成本；
2. 减少不稳定因素；
3. 优先验证岗位源和看板价值。

## Phase 2

低频使用 LLM。

场景：

1. 用户上传简历时，解析一次；
2. 新增岗位时，生成摘要；
3. Top 推荐岗位生成推荐理由；
4. 用户点开岗位时，可按需生成分析。

原则：

1. 只对新增或被用户关注的岗位调用；
2. 结果写入数据库缓存；
3. 不重复调用；
4. 不让 LLM 阻塞抓取主流程；
5. LLM 不编造薪资、截止日期、岗位要求。

## LLM 不做的事

1. 不自动投递；
2. 不登录官网；
3. 不绕验证码；
4. 不直接决定所有匹配分；
5. 不生成未经来源支持的薪资或截止日期。

## 13. 合规与安全

## 13.1 抓取合规

Crawler 必须遵守：

1. 只抓公开页面；
2. 不登录；
3. 不绕验证码；
4. 不破解接口；
5. 不使用代理池；
6. 遇到 robots 禁止则停止；
7. 遇到 403/429 则停止该源；
8. 请求频率低；
9. 不抓取 HR 私人联系方式；
10. 不复刻完整 JD。

## 13.2 用户数据安全

用户数据包括：

1. 邮箱；
2. 偏好；
3. 岗位操作；
4. 后续可能有简历。

Phase 1 不上传简历，因此隐私风险较低。

Phase 2 如果加入简历：

1. 原始简历应可删除；
2. 不应写入日志；
3. 不应对其他用户可见；
4. 解析结果应允许用户编辑和删除。

## 16. 第一阶段验收标准

### 系统验收

1. Vercel 上能访问看板；
2. Supabase Auth 登录可用；
3. 至少 3 个用户可以登录；
4. 6 张表创建完成；
5. RLS 生效；
6. GitHub Actions 能手动触发；
7. GitHub Actions 能定时触发。

### 数据验收

1. 至少 5 家 sources 存在；
2. 至少 3 家源能成功抓取；
3. jobs 表有岗位数据；
4. 重复抓取不会重复插入；
5. crawl_runs 能记录成功和失败。

### 用户体验验收

1. 用户能设置 preferences；
2. 用户看到按自己偏好排序的岗位；
3. 用户可以保存岗位；
4. 用户可以忽略岗位；
5. 用户可以标记已投递；
6. 不同用户操作互不影响；
7. 岗位卡片可以点击跳转官网。

### 成本验收

1. Phase 1 不调用 LLM；
2. 不使用付费服务器；
3. 不自托管模型；
4. 月成本控制在 0–100 元。

## 17. 给开发 Agent 的执行原则

开发 Agent 必须遵守：

1. 不要扩大范围；
2. 不要提前实现商业化功能；
3. 不要引入重型架构；
4. 不要添加自动投递；
5. 不要抓第三方招聘平台；
6. 不要绕过验证码；
7. 不要把 service role key 暴露到前端；
8. 优先跑通端到端闭环；
9. 每完成一个阶段，输出变更说明；
10. 每次修改后运行基础检查。

建议每次任务结束时输出：

1. 修改了哪些文件；
2. 新增了哪些功能；
3. 如何本地运行；
4. 如何验证；
5. 还有哪些未完成；
6. 下一步建议。

## 18. 总结

本技术方案的核心思想是：

> 用最轻的技术栈搭建一个可用的 3–5 人岗位雷达看板，先验证真实使用价值，再逐步加入 LLM、邮件、更多源和商业化功能。

当前阶段最重要的不是 AI，也不是复杂架构，而是：

1. 每天能稳定拿到岗位；
2. 用户能方便地看岗位；
3. 用户能记录自己的动作；
4. 不同用户偏好和状态能隔离；
5. 成本足够低。
