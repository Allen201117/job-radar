# 求职雷达核心链路复盘

日期：2026-05-26

## 当前判断

此前 PoC 的主要问题不是 UI，而是数据链路不可信：

- 登录被 `AutoLogin` 旁路；
- 搜索混入非 Phase 1 source；
- 百度、京东、海尔、Siemens 把导航/入口页写成岗位；
- `location = null` 时去重不稳定；
- 用户操作无法覆盖实时搜索结果，因为实时结果没有稳定 DB id。

## 已修正的链路

当前链路改为：

```text
Apple / Siemens 官方页面 crawler
  -> 质量闸门校验 jd_url
  -> jobs active 入库或更新
  -> Jobs / Today 读取
  -> 用户 saved / ignored / applied
  -> 点击企业官方详情页

百度 / 京东已知中国官网源刷新
  -> 质量闸门校验 jd_url
  -> 服务端 upsert 到 jobs
  -> 前端使用稳定 DB id 支持用户操作

百度千帆动态官方源发现
  -> 用户点击触发，默认只调用 1 个 generated query
  -> 45 分钟内相同 user/query/city/job_type 复用缓存
  -> 官方源候选先写 source_candidates
  -> 只有 parser + jd_url 质量门通过才写 jobs

简历画像
  -> 文本 / .txt / .md 解析 candidate profile
  -> 用户确认后同步 user_preferences
  -> Jobs 排序受偏好变化影响
```

## 实时搜索边界

当前必须区分三条路径：

1. 搜索已有岗位：只筛本地 `jobs` 表，不触发外部请求。
2. 刷新已知中国官网源：走 `/api/search`，只访问百度 / 京东已验证官方公开源，不消耗百度千帆额度。
3. 发现中国官方招聘源：走 `/api/discovery`，百度千帆是主 provider，但只能低频串行调用。

百度千帆“百度搜索”免费额度为每日 50 次。控制台显示剩余 0/50 或未开通付费时，设置：

```bash
BAIDU_QIANFAN_SEARCH_DISABLED=true
```

开启后不调用真实千帆搜索，直接返回 `provider_rate_limited` / `rate_limited=true`，用于前端稳定展示和提交前回归。

原因：

- 百度官方社招页 SSR 数据包含 `postId`，官方前端点击逻辑为 `/jobs/detail/{recruitType}/{postId}`；
- 百度抽样详情页可打开且标题匹配；
- 京东公开列表 API 可返回 `requirementId`，详情页为 `zhaopin.jd.com/web/job-info-detail?requementId=...`；
- 服务端可将 known source 结果 upsert 到 `jobs`，因此用户操作不会断链。

暂不启用：

- 非首批 source；
- 第三方招聘平台；
- 无稳定详情 URL 的 source；
- 需要登录、验证码、403/429 的路径。

## 剩余风险

- 当前 Codex 沙箱可能禁止监听本地端口，前端手点验证可能需要用户本机直接执行 `npm run dev`；
- 当前环境网络审批可能阻断 Supabase / 百度 / 京东 live 回归；被阻断时不能把本地单测冒充 live 验证；
- 百度千帆额度耗尽时不要继续跑 dynamic discovery live query；
- Moka SPA hash detail 仍需证明可得到服务端可验证的真实详情 URL；
- 高校就业转载页、第三方招聘平台、公众号/知乎/SEO 聚合页不能进入 `source_candidates` 官方源成功证据；
- 如果某个 source 只能拿列表不能拿详情页，必须保持 `partial_success`；
- 不应为了看起来数据多而恢复低质量 source。

## 下一步

1. 等百度千帆额度恢复后，只先跑 2 个 query，每个 query 只调用 1 个 generated query。
2. 若没有 429，再串行测试剩余 query，保持间隔，记录 raw results 前 10 条。
3. 从已有 `source_candidates` 里优先找非 Moka、非高校转载、非 SPA hash-only 的官方详情页，补 generic parser。
4. 继续保持 `jobs` 写入质量门：官方详情页、HTTP 200、title 或核心片段匹配。
