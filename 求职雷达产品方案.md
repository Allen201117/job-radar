# Job Radar Private Beta 产品方案

版本：v0.1  
日期：2026-05-18  
阶段：3–5 人轻量内测版  
产品定位：低成本官网岗位雷达看板  
当前边界：不做自动投递、不登录企业招聘系统、不绕过验证码、不抓取第三方招聘平台、不抓取登录后内容、不复刻完整 JD

---

## 0. 文档目的

本文档用于指导开发 Agent 搭建 **Job Radar Private Beta v0.1**。

本项目当前不是商业化 SaaS，也不是单人本地脚本，而是一个给 3–5 个早期用户使用的轻量级官网岗位雷达看板。

产品目标是：

1. 每天低频监控一批企业官网招聘页面；
2. 将公开岗位信息整理到共享岗位库；
3. 每个用户登录后根据自己的偏好看到不同排序；
4. 用户可以收藏、忽略、标记已投递；
5. 点击岗位详情直接跳转官网；
6. 先验证真实使用价值，再考虑后续商业化。

---

## 1. 一句话产品定义

**Job Radar 是一个面向小范围早期用户的官网岗位雷达看板：系统每天自动检查公开企业招聘官网，把新增岗位统一整理入库，并根据用户偏好生成个人化岗位列表，帮助用户减少手动检索成本，提高投递决策效率。**

---

## 2. 为什么要做这个产品

求职者现在面临的问题不是“没有招聘信息”，而是“信息过于分散、重复检索成本过高、官网岗位容易错过”。

典型痛点包括：

1. 用户每天需要手动打开多个企业官网；
2. 不同企业官网结构不同，校招、社招、实习入口分散；
3. 第三方招聘平台不一定比官网更新快；
4. 外企 careers 页面经常是英文，筛选成本高；
5. 用户不知道哪些岗位是新增的；
6. 用户没有统一地方记录“已收藏、已投递、已忽略”；
7. 用户转行时需要关键词探索，而不是只按一个固定岗位搜索；
8. 手动维护投递表格非常耗时。

本项目不是替用户自动投递，而是先解决：

> “我今天应该看哪些官网岗位？”

---

## 3. 当前产品形态

当前产品应被定义为：

**Private Beta / 小范围内测版**

目标用户：

1. 产品发起人本人；
2. 身边 3–5 个真实求职用户；
3. 这些用户可以有不同求职方向，例如数据分析、商业分析、产品经理、战略/咨询、金融研究、外企管培、实习、校招、entry-level 社招。

产品形态：

1. 一个 Vercel 部署的 Web 看板；
2. 用户使用 Supabase Auth 登录；
3. 所有用户共享同一个岗位库；
4. 每个用户有自己的偏好和岗位操作记录；
5. 每天由 GitHub Actions 自动运行 crawler；
6. 第一阶段不做移动 App，不做飞书/微信推送。

---

## 4. 核心产品原则

### 4.1 低成本优先

当前阶段成本目标是：

- Phase 1：0–100 元/月；
- Phase 2：50–150 元/月；
- 商业化前不应超过 300 元/月。

因此当前不使用：

1. GPU；
2. 自托管大模型；
3. K8s；
4. Redis；
5. Celery；
6. Prometheus/Grafana/Loki；
7. 复杂多租户计费系统。

### 4.2 合规边界优先

系统只访问公开企业招聘页面。

明确不做：

1. 不自动投递；
2. 不登录企业招聘系统；
3. 不绕过验证码；
4. 不抓取登录后内容；
5. 不抓取第三方招聘平台；
6. 不抓取 HR 私人联系方式；
7. 不复刻完整 JD；
8. 不使用代理池规避封禁；
9. 不破解接口签名或加密参数。

如果某个源出现以下情况，应立即停止抓取并记录：

1. 需要登录；
2. 出现验证码；
3. 返回 403/429；
4. robots.txt 明确禁止；
5. 跳转到登录页；
6. 页面明显是反爬提示。

### 4.3 共享岗位库，用户操作隔离

岗位源只抓一次，所有用户共享岗位库。

但每个用户的以下内容必须隔离：

1. 偏好设置；
2. 收藏；
3. 忽略；
4. 已投递；
5. 浏览记录；
6. 备注。

同一个岗位可以同时被用户 A 标记“已投递”，被用户 B 标记“收藏”。

### 4.4 先做看板，不先做复杂 AI

第一阶段先用规则匹配。LLM 不作为核心依赖。

LLM 后续可以用于：

1. 简历解析；
2. 岗位摘要；
3. 推荐理由；
4. 技能标签提取；
5. 简历与岗位差距说明。

但第一阶段重点是跑通：

> 官网岗位源 → 岗位入库 → 用户看板 → 官网跳转 → 用户反馈

---

## 5. 用户角色

### 5.1 普通用户

普通用户可以：

1. 登录系统；
2. 查看岗位看板；
3. 设置自己的求职偏好；
4. 按公司、城市、关键词筛选岗位；
5. 收藏岗位；
6. 忽略岗位；
7. 标记已投递；
8. 点击官网链接查看完整 JD 或投递。

普通用户不可以：

1. 新增或修改 source；
2. 手动触发全局 crawler；
3. 查看其他用户的操作；
4. 查看管理员日志。

### 5.2 管理员

管理员通常是产品发起人本人。

管理员可以：

1. 新增企业招聘源；
2. 启用/禁用 source；
3. 查看抓取日志；
4. 查看每个 source 最近抓取状态；
5. 手动触发 crawler；
6. 管理早期用户；
7. 查看系统错误信息。

---

## 6. 产品分阶段迭代路线

## Phase 0：Source PoC

目标周期：3–5 天  
用户范围：仅产品发起人  
成本目标：0 元

### 目标

验证公开企业招聘官网是否能被低频、合规地抓取并整理成岗位数据。

### 首批 5 个源

建议先做：

1. 百度；
2. 京东；
3. 海尔；
4. Apple；
5. Siemens。

### 功能范围

只做：

1. sources 配置；
2. crawler 手动运行；
3. 抓取岗位列表；
4. 标准化基础字段；
5. 写入 Supabase；
6. 识别新增岗位；
7. 记录抓取日志。

不做：

1. 登录系统；
2. 完整看板；
3. 用户偏好；
4. 收藏/忽略/已投递；
5. LLM；
6. 邮件/飞书/微信推送。

### 验收标准

1. 5 家源里至少 3 家能抓取成功；
2. 每家源能抓到 20 条岗位，或该源当前全部公开岗位；
3. 每条岗位至少有公司、岗位名、城市或地点、官网链接、首次发现时间；
4. 重复运行不会重复插入相同岗位；
5. 抓取失败能写入 crawl_runs。

## Phase 1：Private Beta v0.1

目标周期：1–2 周  
用户范围：3–5 个邀请制用户  
成本目标：0–100 元/月

### 目标

搭建一个可登录、可查看、可筛选、可记录操作的轻量级岗位看板。

### 必须实现功能

#### 1. 登录

使用 Supabase Auth。

第一阶段可以只支持：

1. 邮箱登录；
2. 手动邀请用户；
3. 管理员手动创建用户或白名单。

#### 2. 岗位看板

用户登录后进入 Today / Jobs 页面，看到共享岗位库。

看板支持：

1. 显示岗位列表；
2. 按匹配分排序；
3. 按 first_seen_at 显示新增；
4. 点击岗位跳转官网。

#### 3. 用户偏好

每个用户可以设置：

1. 目标城市；
2. 目标岗位；
3. 目标关键词；
4. 排除关键词；
5. 关注公司；
6. 每日展示数量。

#### 4. 用户操作

用户可以对岗位执行：

1. viewed；
2. saved；
3. ignored；
4. applied。

这些操作写入 job_actions，并按 user_id 隔离。

#### 5. 简单规则匹配

系统根据用户偏好给岗位打分。

推荐初始规则：

- 标题命中目标岗位：+30；
- 城市命中目标城市：+20；
- 公司命中关注公司：+15；
- 标题或摘要命中目标关键词：每个 +5；
- 岗位是 7 天内新增：+10；
- 命中排除关键词：-50；
- 已 ignored：不展示；
- 已 applied：默认隐藏，但可筛选查看。

#### 6. 管理员 source 页面

管理员可以查看：

1. company；
2. source_url；
3. enabled；
4. crawl_method；
5. last_checked_at；
6. notes。

管理员可以启用/禁用 source。

#### 7. GitHub Actions 每日抓取

每天早上 8 点新加坡/中国时间自动运行 crawler。

也支持手动 workflow_dispatch。

### 不做功能

Phase 1 不做：

1. 自动投递；
2. 第三方招聘平台抓取；
3. 微信/飞书推送；
4. 邮件 digest；
5. 简历上传；
6. LLM 摘要；
7. LLM 匹配；
8. 复杂推荐模型；
9. 复杂监控；
10. 付费系统；
11. 商业化多租户。

### Phase 1 验收标准

1. Vercel 上能打开网站；
2. 用户可以通过 Supabase Auth 登录；
3. 至少 3 个测试用户可以分别设置偏好；
4. 所有用户共享同一个 jobs 表；
5. 同一个 jobs 表能按不同用户偏好显示不同排序；
6. saved / ignored / applied 操作按用户隔离；
7. GitHub Actions 能定时和手动触发 crawler；
8. 至少 5 家源可以抓取并写入 jobs；
9. 岗位卡片可以跳转官网；
10. 月成本控制在 0–100 元。

## Phase 2：Private Beta v0.2

目标周期：2–4 周  
用户范围：5–10 个内测用户  
成本目标：50–150 元/月

### 目标

在看板稳定后，加入低频 LLM 能力，提高岗位理解与推荐解释质量。

### 新增功能

#### 1. 简历上传与解析

用户上传 PDF 或 DOCX 简历。

LLM 提取：

1. 教育背景；
2. 专业；
3. 技能；
4. 实习；
5. 项目；
6. 目标岗位关键词；
7. 可迁移能力。

用户必须可以手动修改解析结果。

#### 2. 岗位摘要

只对以下岗位调用 LLM：

1. 新增岗位；
2. 用户点开的岗位；
3. Top 推荐岗位。

摘要生成后缓存，不重复生成。

#### 3. 推荐理由

对 Top 推荐岗位生成解释，例如：

1. 命中哪些技能；
2. 哪些要求缺失；
3. 为什么建议优先看；
4. 简历应如何调整。

#### 4. 邮件 digest

每天发一封邮件，内容包括：

1. 今日新增；
2. 高匹配岗位；
3. 收藏未投递岗位；
4. 官网跳转链接。

### Phase 2 验收标准

1. 用户可以上传简历；
2. LLM 能提取基本简历关键词；
3. 新增岗位能生成摘要；
4. 推荐理由可读、可解释、不编造；
5. LLM 结果缓存；
6. 月成本控制在 50–150 元。

## Phase 3：Alpha v0.3

目标周期：1–2 个月  
用户范围：10–30 个用户  
成本目标：100–300 元/月

### 目标

扩大数据源，增强用户反馈和管理能力。

### 新增功能

1. 数据源扩展到 30–50 家；
2. source health 简单页面；
3. 更详细的 crawl error 分类；
4. 用户反馈原因；
5. 推荐排序根据反馈微调；
6. 支持导出 CSV；
7. 支持更多筛选，包括新增时间、薪资是否披露、校招/社招/实习、是否收藏、是否已投递。

## Phase 4：商业化预备版

只有在以下条件成立后才进入：

1. 早期用户连续使用 2–4 周；
2. 用户反馈明确认为节省时间；
3. source 维护成本可控；
4. 用户愿意推荐给其他人；

可能新增：

1. 多用户邀请系统；
2. 付费订阅；
3. 更完整隐私与合规；
4. 更多 source；
5. 飞书/微信推送；
6. 高级简历匹配；
7. 岗位横向比较；
8. 校招批次日历；
9. 更完整监控；
10. 更严格权限体系。

## 8. 数据展示规则

### 8.1 岗位摘要

Phase 1 不使用 LLM 摘要。

摘要来源优先级：

1. adapter 提取到的简短描述；
2. JD 前 200 字截断；
3. 标题 + 公司 + 城市组成简短说明。

### 8.2 薪资展示

薪资只展示官网明确披露的文本。

如果未披露，显示：

> 官网未披露

不要估算薪资。

### 8.3 官网链接

每张岗位卡片必须有 “查看官网” 按钮。

用户点击后跳转到 jd_url 或 apply_url。

### 8.4 已忽略岗位

默认不展示已忽略岗位。

用户可以通过筛选查看。

### 8.5 已投递岗位

默认不展示在推荐列表中。

用户可以在 Applied 页面查看。

---

## 9. LLM 在产品中的作用

LLM 不是第一阶段核心依赖。

后续 LLM 主要用于：

1. 解析简历；
2. 提取技能；
3. 生成岗位摘要；
4. 生成推荐理由；
5. 识别岗位与简历的差距；
6. 生成简历修改建议。

LLM 不用于：

1. 自动投递；
2. 登录官网；
3. 绕过验证码；
4. 编造薪资；
5. 编造截止时间；
6. 单独决定最终匹配分。

推荐原则：

> 规则打分负责排序，LLM 负责读懂与解释。

---

## 10. 成功指标

Phase 1 重点观察：

1. 用户是否每天打开看板；
2. 用户是否点击官网链接；
3. 用户是否标记 saved / ignored / applied；
4. 每天新增岗位是否稳定；
5. 是否出现大量无关岗位；
6. source 是否频繁失效；
7. 用户是否觉得节省时间。

建议内测期间每周人工问用户：

1. 今天的推荐岗位有没有用？
2. 是否比自己手动搜更省时间？
3. 哪些岗位明显不相关？
4. 希望增加哪些公司？
5. 是否愿意继续使用？

---

## 11. 当前阶段最终判断

当前最佳路线不是商业化 SaaS，也不是本地脚本，而是：

**Vercel + Supabase + GitHub Actions 的 3–5 人 Private Beta。**

先验证最小闭环：

> 公开官网岗位 → 共享岗位库 → 用户偏好排序 → 岗位看板 → 官网跳转 → 用户反馈

如果这个闭环成立，再逐步加入 LLM、邮件、飞书/微信和商业化能力。
